"机制很多, 循环一个" -- 全部 19 章机制归于同一个 while True。

Harness 层: 综合 -- 工具、权限、记忆、任务、团队、插件都挂在同一个循环上。

```python

不是再发明新机制，而是把前 19 章的教学组件合成一个完整 harness。
一个 while True 承载全部：cron 注入 → 上下文预算 → LLM 调用 → 权限闸门 →
工具分发（内置+MCP+后台）→ tool_result 回传 → 下一轮。

  用户输入

            → TOOL_HANDLERS / MCP handlers / background dispatch
```

### 💡 核心设计模式解读：

1. **从 s19 到 s20：补齐了哪些教学组件？**
   | 组件 | s19 | s20 |
   |------|-----|-----|
   | 工具池 | 内置 + MCP | 内置 + MCP，补齐 s01-s18 全部工具（27 个） |
   | 权限 | 教学主体省略 | `PreToolUse` hook 中执行（permission_hook） |
   | hooks | 省略 | UserPromptSubmit / PreToolUse / PostToolUse / Stop 全部注册 |
   | todo | 省略 | `todo_write` + rounds_since_todo 提醒（>=3 轮 nag） |
   | skill | 省略 | catalog in system prompt + `load_skill` |
   | compact | 省略 | 4 层管道：tool_result_budget → snip → micro → compact |
   | error recovery | 简化 try/except | `with_retry`（429/529）+ `max_tokens` 升级 + `reactive_compact` |
   | background | 省略 | 慢操作后台线程 + `<task_notification>` 注入 |
   | cron | 省略 | daemon scheduler + durable jobs + cron_autorun_loop |
   | subagent | 省略 | `spawn_subagent`：独立 messages[], 30 轮限制, 只返回摘要 |

   s20 不是 s19 的增量修改——它是**把 s01-s18 中被 s19 省略的模块全部放回来的综合版**。27 个内置工具 vs s19 的 ~19 个，4 层压缩管道 vs 无，完整的权限+hooks+错误恢复 vs 简化版。

2. **agent_loop 的 7 步流水线：每一轮 LLM 调用前的固定仪式**
   ```
   ① cron/background 注入        → messages 增加 "[Scheduled]" 或 task_notification
   ② todo 提醒                    → >=3 轮未更新时注入 "<reminder>Update your todos</reminder>"
   ③ 上下文预算管道               → 4 层压缩确保 messages 不超限
   ④ assemble_tool_pool + system   → 动态工具池 + 实时 context 拼装
   ⑤ call_llm (with_retry)        → 带错误恢复的 LLM 调用
   ⑥ 遍历 tool_use blocks          → 权限闸门 → 分发 → 执行 → hooks
   ⑦ build_user_content → messages → 合并 tool_result + background通知
   ```
   每一步都是一个独立函数，互不耦合。新增机制可以挂在任意位置——
   例如权限挂在步骤⑥的 `PreToolUse` hook 上，压缩挂在步骤③的管道里。

3. **两层计划系统共存：todo_write + task graph**
   | | todo_write (s05) | task graph (s12) |
   |---|---|---|
   | 存储 | 进程内存 `CURRENT_TODOS` | `.tasks/{id}.json` 文件 |
   | 生命周期 | 当前会话（退出即消失） | 跨会话持久化 |
   | 依赖 | 无 | `blockedBy` DAG 依赖 |
   | 协作 | 单人 | `owner` 字段支持多 Agent 认领 |
   | 用途 | "我现在要干嘛" | "这个项目要干嘛" |

   s20 同时保留两层：`todo_write` 帮助单个 Agent 不漂移，task graph 支撑 team 协作。`rounds_since_todo` 计数器追踪"多久没更新 todo了"——>=3 轮时自动注入提醒。

4. **两种 delegation：subagent vs teammate**
   | | subagent (s06) | teammate (s15-s17) |
   |---|---|---|
   | 启动方式 | `spawn_subagent(description)` | `spawn_teammate_thread(name, role, prompt)` |
   | 上下文 | 独立 `messages[]`，30 轮限制 | 独立 `messages[]`，WORK/IDLE 周期 |
   | 返回 | 仅返回最终文本摘要 | 通过 MessageBus 收发，可多轮通信 |
   | 并发 | 同步等待 | 后台线程并行 |
   | 用途 | "上下文隔离" | "长期并行协作" |
   | 工具 | bash/read/write/edit/glob (5 个) | bash/read/write/send/submit_plan/list/claim/complete (8 个) |

   s20 的 teammate 新增了 `protocol_ctx["waiting_plan"]`——提交计划后暂停所有模型步骤，直到 Lead 发 `plan_approval_response`。这是真正的代码级门禁，不是依赖模型自觉。

5. **4 层上下文压缩管道：便宜优先，昂贵最后**
   ```
   Layer 1: tool_result_budget  → 最后一轮 tool_result 总大小 > 200KB?
                                   优先持久化最大的输出到文件
   Layer 2: snip_compact         → 消息数 > 50? 保留头3 + 尾47，中间标记 snipped
   Layer 3: micro_compact        → tool_result 超过 3 条? 旧结果改为占位符
   Layer 4: compact_history      → 总大小 > 50KB? 保存 transcript → LLM 摘要
   ```
   从 s08 继承的分层策略：前 3 层是纯字符串操作（免费），第 4 层才调 LLM（昂贵）。`compact` 工具被 agent_loop 截获——不经过权限/handler，直接执行压缩后 continue。

6. **错误恢复三条路径：升级 → 压缩 → 重试**
   ```
   路径 1 (max_tokens):      stop_reason == "max_tokens"?
                               → max_tokens 8000 → 16000（一次升级机会）
                               → 仍不足? 注入 CONTINUATION_PROMPT（最多 2 次）
   路径 2 (prompt too long):  prompt_length_exceeded?
                               → reactive_compact（保存 transcript + LLM 摘要）
                               → 保留最近 5 条消息 + 摘要
   路径 3 (429/529):          with_retry:
                               429: 指数退避重试（最多 3 次）
                               529: 连续 2 次 → 切 FALLBACK_MODEL
   ```
   三条路径互不干扰。`RecoveryState` 追踪所有恢复状态：`has_escalated`、`recovery_count`、`consecutive_529`、`has_attempted_reactive_compact`、`current_model`。

7. **权限闸门：三道防线挂在 PreToolUse hook 上**
   ```python
   def permission_hook(block):
       # 防线 1: bash 黑名单
       if block.name == "bash":
           for pattern in DENY_LIST:       # "rm -rf /", "sudo", ...
               if pattern in command: deny
           if any(token in command for token in DESTRUCTIVE):  # "rm ", "chmod 777"
               user_confirm() → allow/deny

       # 防线 2: write_file 路径逃逸
       if block.name in ("write_file", "edit_file"):
           safe_path(path) → 校验不通过? deny

       # 防线 3: MCP destructive 工具
       if block.name.startswith("mcp__") and "deploy" in block.name:
           user_confirm() → allow/deny
   ```
   权限不写死在工具执行代码里，而是作为 hook 挂在 PreToolUse 上。好处：新增权限规则只需注册新 hook，不改工具函数。`log_hook` 也在同一事件上注册，和权限完全独立。

8. **线程模型：四种并发角色**
   | 线程 | 角色 | 锁 |
   |------|------|-----|
   | 主线程 | 用户交互 + agent_loop | 持 `agent_lock` |
   | `cron_scheduler_loop` | 定时检查 cron 表达式 + 投递到 cron_queue | `cron_lock` |
   | `cron_autorun_loop` | 检测 cron_queue → 持 `agent_lock` 调用 agent_loop | `agent_lock`（blocking） |
   | `background_tasks` workers | 执行慢 bash 命令 | `background_lock` |
   | `teammate` threads | 独立 agent_loop | 无全局锁（通过 MessageBus 通信） |

   `agent_lock` 确保同一时间只有一个 agent_loop 在运行——用户输入和 cron 自动运行互斥。`cron_scheduler_loop` 只写 `cron_queue`（持 `cron_lock`），不碰 `agent_lock`。

9. **工具池全景：27 个内置工具 + MCP 动态扩展**
   ```text
   基础工具 (5):  bash, read_file, write_file, edit_file, glob
   计划工具 (2):  todo_write, task (subagent)
   技能 (2):      load_skill, compact
   任务图 (5):    create_task, list_tasks, get_task, claim_task, complete_task
   定时 (3):      schedule_cron, list_crons, cancel_cron
   团队 (6):      spawn_teammate, send_message, check_inbox,
                 request_shutdown, request_plan, review_plan
   Worktree (3):  create_worktree, remove_worktree, keep_worktree
   MCP (1):       connect_mcp
   ─────────────────────────────────────
   合计: 27 内置 + N 个已连接 MCP 工具
   ```
   每个工具都有对应的 handler（在 `BUILTIN_HANDLERS` 中），`assemble_tool_pool()` 每轮动态合并 MCP 工具。

10. **s19 → s20：全面补齐，而非增量修改**
    | 新增/补齐 | 说明 |

    | `terminal_print()` | 线程安全的终端打印（后台线程不覆盖用户输入行） |
    | `CURRENT_TODOS` + `run_todo_write()` | s05 的 todo 计划系统回归 |
    | `SKILL_REGISTRY` + `scan_skills()` + `list_skills()` + `load_skill()` | s07 的技能加载系统回归 |
    | `compact_history()` + 3 层辅助函数 | s08 的 4 层压缩管道回归 |
    | `RecoveryState` + `with_retry()` + `is_prompt_too_long_error()` | s11 的错误恢复系统回归 |
    | `is_slow_operation()` + `should_run_background()` + `start_background_task()` + `collect_background_results()` | s13 的后台任务系统回归 |
    | `CronJob` + cron 全套 (12 个函数) + `cron_autorun_loop()` | s14 的 cron 调度系统回归 |
    | `spawn_subagent()` + `SUB_TOOLS` + `SUB_HANDLERS` | s06 的子 Agent 系统回归 |
