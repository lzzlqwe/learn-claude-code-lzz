"工具不够用? 插个外挂" -- MCP 客户端 + 工具发现 + 动态组装统一工具池。

Harness 层: 插件 -- 外部工具按需接入, 名称安全规范化, 运行时动态合并。

```python
s19_mcp_plugin/code.py - 带 MCP 插件系统的 Agent

在 s18 的 Worktree 隔离之上，新增 MCP（Model Context Protocol）插件系统。
Agent 可以连接外部 MCP 服务器（docs/deploy），动态发现工具并合并到统一工具池。
工具名经过 normalize 安全处理，格式为 mcp__{server}__{tool}，运行时组装。

3 大 MCP 机制:
  MCPClient: 封装单个 MCP 服务器的工具注册与调用（教学版用 mock handler）
```

### 💡 核心设计模式解读：

1. **s18 静态工具 → s19 动态工具池：改了什么？**
   | | s18 及之前 | s19 MCP |
   |---|---|---|
   | 工具列表 | `TOOLS = [...]` 写死在代码里 | `assemble_tool_pool()` 运行时组装 |
   | 工具来源 | 仅内置工具（bash/read/write/...） | 内置 + 已连接 MCP 服务器的工具 |
   | 扩展方式 | 改代码 → 重启 | `connect_mcp("server")` → 即时可用 |
   | System Prompt | 固定模板 | 动态注入 "Connected MCP servers: docs, deploy" |
   | Handler 分发 | `TOOL_HANDLERS` 字典 | `BUILTIN_HANDLERS` + MCP handler 闭包合并 |

   核心变化：工具从"编译时常量"变成"运行时可扩展"。`assemble_tool_pool()` 每次返回的 `tools` 和 `handlers` 取决于当前已连接的 MCP 服务器——连了 docs 就有 search/get_version，连了 deploy 就有 trigger/status。

2. **MCPClient 设计：注册 + 调用分离**
   ```python
   class MCPClient:
       def __init__(self, name):
           self.name = name
           self.tools = []              # 工具定义（schema）→ 发给 LLM
           self._handlers = {}          # 工具 handler（callable）→ 执行

       def register(self, tool_defs, handlers):
           self.tools = tool_defs       # LLM 看到的内容
           self._handlers = handlers    # Agent 执行的函数

       def call_tool(self, tool_name, args):
           handler = self._handlers.get(tool_name)
           if not handler: return "MCP error: unknown tool"
           return handler(**args)
   ```
   `tools` 和 `_handlers` 是一一对应的两套数据：
   - `tools`：发给 LLM 的 JSON Schema（描述工具名、参数、是否 readOnly）
   - `_handlers`：Python 可调用对象（真正执行操作的函数）

   真实 MCP 协议中，`tools` 来自服务器的 `tools/list` 响应，`call_tool` 通过 `tools/call` 消息通信。教学版跳过协议层，直接 mock。

3. **normalize_mcp_name：名称安全的第二道防线**
   ```python
   _DISALLOWED_CHARS = re.compile(r'[^a-zA-Z0-9_-]')

   def normalize_mcp_name(name: str) -> str:
       return _DISALLOWED_CHARS.sub('_', name)
   ```
   为什么需要？因为工具名会变成 `mcp__{server}__{tool}` 格式传给 LLM。如果服务器名或工具名包含特殊字符（空格、斜杠、中文），会破坏工具名的解析。

   `normalize_mcp_name` 使用和 `validate_worktree_name` 类似的白名单策略——只保留 `[a-zA-Z0-9_-]`，其余全部替换为 `_`。这是在"信任外部工具输入"和"保护内部系统"之间的最小化安全边界。

4. **assemble_tool_pool：运行时合并内置 + MCP 工具**
   ```python
   def assemble_tool_pool():
       tools = list(BUILTIN_TOOLS)          # ① 从内置工具开始
       handlers = dict(BUILTIN_HANDLERS)

       for server_name, mcp_client in mcp_clients.items():  # ② 遍历所有已连接服务器
           safe_server = normalize_mcp_name(server_name)
           for tool_def in mcp_client.tools:
               safe_tool = normalize_mcp_name(tool_def["name"])
               prefixed = f"mcp__{safe_server}__{safe_tool}"  # ③ 拼装唯一工具名

               tools.append({...prefixed...})
               handlers[prefixed] = (                           # ④ 闭包绑定
                   lambda *, c=mcp_client, t=tool_def["name"], **kw:
                       c.call_tool(t, kw))
       return tools, handlers
   ```
   四个关键设计：
   - **`list()` + `dict()` 拷贝**：不修改 `BUILTIN_TOOLS` 和 `BUILTIN_HANDLERS` 原对象，每次返回独立副本
   - **`mcp__` 前缀**：工具名上加命名空间，`mcp__docs__search` 不会和内置 `search` 工具冲突
   - **闭包默认参数陷阱**：`lambda *, c=mcp_client, t=tool_def["name"]` —— 必须用默认参数捕获当前循环变量值，否则所有闭包都引用最后一次迭代的值（经典 Python 闭包延迟绑定问题）
   - **每次重新组装**：connect_mcp 后重新调用，新工具即时可用

5. **动态刷新：connect_mcp 后的工具池重建**
   ```python
   def agent_loop(messages, context):
       tools, handlers = assemble_tool_pool()      # 初始组装
       system = assemble_system_prompt(context)
       while True:
           response = client.messages.create(..., tools=tools)  # 用当前工具池
           # ... 执行工具 ...
           
           # 检测到 connect_mcp 调用 → 刷新
           if any(b.name == "connect_mcp" for b in response.content
                  if b.type == "tool_use"):
               tools, handlers = assemble_tool_pool()  # 重新组装
               context = update_context(context, messages)
               system = assemble_system_prompt(context)  # 重建 SYSTEM
   ```
   刷新触发条件：本轮 LLM 响应中包含了 `connect_mcp` 工具调用。刷新后：
   - `tools` 包含新连接的 MCP 服务器的工具
   - `handlers` 包含新工具的 handler
   - `system` prompt 更新为包含新 MCP 服务器名

   注意 s19 去掉了 prompt cache——因为工具池动态变化，缓存的 SYSTEM prompt 可能在连接 MCP 后过期。这是正确性优先于性能的取舍。

6. **Mock 服务器：教学版 MCP 协议模拟**
   ```python
   def _mock_server_docs():
       client = MCPClient("docs")
       client.register(
           tool_defs=[
               {"name": "search", "description": "Search documentation. (readOnly)", ...},
               {"name": "get_version", "description": "Get API version. (readOnly)", ...},
           ],
           handlers={
               "search": lambda query: f"[docs] Found 3 results for '{query}'",
               "get_version": lambda: "[docs] API v2.1.0",
           })
       return client

   MOCK_SERVERS = {"docs": _mock_server_docs, "deploy": _mock_server_deploy}
   ```
   两个 mock 服务器的工具对比：
   | 服务器 | 工具 | 类型标注 | 说明 |
   |-------|------|---------|------|
   | docs | search | readOnly | 搜索文档 |
   | docs | get_version | readOnly | 获取 API 版本 |
   | deploy | trigger | destructive | 触发部署（真实 CC 需审批） |
   | deploy | status | readOnly | 检查部署状态 |

   `(readOnly)` 和 `(destructive)` 标注告诉 LLM 工具的安全级别。真实 CC 中，destructive 工具会触发权限闸门（s03）。

7. **BUILTIN_TOOLS/BUILTIN_HANDLERS 分离：为动态组装做的重构**
   ```python
   # s18: 单一 TOOLS 列表 + TOOL_HANDLERS 字典
   TOOLS = [bash, read, write, ...]
   TOOL_HANDLERS = {"bash": run_bash, ...}

   # s19: BUILTIN_TOOLS + BUILTIN_HANDLERS → assemble_tool_pool() 合并
   BUILTIN_TOOLS = [bash, read, write, ...]
   BUILTIN_HANDLERS = {"bash": run_bash, ...}
   ```
   s18 的 `TOOLS`/`TOOL_HANDLERS` 是全局常量，agent_loop 直接使用。s19 重命名为 `BUILTIN_TOOLS`/`BUILTIN_HANDLERS`，语义上明确"这只是内置的基础工具"——MCP 工具是附加的。`assemble_tool_pool()` 把它们和 MCP 工具合并为一个动态池。

8. **s18 → s19：加了 1 个类 + 5 个函数 + 工具池重构**
   | 新增 | 说明 |
   |------|------|
   | `MCPClient` class | MCP 客户端（register + call_tool） |
   | `normalize_mcp_name()` | 名称安全规范化（非法字符 → 下划线） |
   | `connect_mcp()` | 连接 MCP 服务器并发现工具 |
   | `assemble_tool_pool()` | 运行时合并内置 + MCP 工具 |
   | `_mock_server_docs()` | mock MCP 服务器：docs |
   | `_mock_server_deploy()` | mock MCP 服务器：deploy |
   | `MOCK_SERVERS` dict | mock 服务器注册表 |
   | `mcp_clients` dict | 已连接的 MCP 客户端实例 |
   | `run_connect_mcp` | connect_mcp 工具处理函数 |
   | `BUILTIN_TOOLS` / `BUILTIN_HANDLERS` | 重构：从 TOOLS/TOOL_HANDLERS 分离 |
   | `assemble_system_prompt` 升级 | 动态注入已连接 MCP 服务器列表 |
   | `agent_loop` 升级 | 每轮 assemble_tool_pool + connect_mcp 后动态刷新 |

   s19 的核心变化不只是"加了 MCP 工具"，而是**把整个工具系统从静态扩展为动态**。从 s02 就开始的 `TOOLS = [...]` 模式在 s19 被彻底打破——工具不再是写死的列表，而是根据运行时状态动态组装的池。
