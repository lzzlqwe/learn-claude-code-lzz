"各干各的, 互不踩脚" -- git worktree 隔离 + 任务-目录绑定 + 事件日志。

Harness 层: 隔离 -- 每个任务独立分支 + 独立工作目录, teammate 自动切换 cwd。

```python
s18_worktree_isolation/code.py - 带 git worktree 隔离的 Agent

在 s17 的自主 Agent 之上，新增 git worktree 隔离系统。每个任务可以绑定到独立的
git worktree（分支 wt/{name} + 目录 .worktrees/{name}），teammate 认领任务后
自动切换 cwd 到绑定的 worktree 目录，bash/read/write 全部在隔离环境中执行。

3 大隔离机制:
  git worktree: git worktree add 创建独立分支和目录，多个 teammate 同时工作互不干扰
  Task.worktree 绑定: 任务 JSON 的 worktree 字段记录绑定名，teammate 自动切换 cwd
  events.jsonl: 记录 worktree 生命周期事件（create/remove/keep），可审计追踪
```

### 💡 核心设计模式解读：

1. **git worktree vs 直接在主仓库工作：隔离了什么？**
   | | 主仓库直接操作 | git worktree 隔离 |
   |---|---|---|
   | 文件层面 | 所有 Agent 操作同一个目录 | 每个 Agent 有独立的 .worktrees/{name}/ 目录 |
   | Git 分支 | 共享一个分支，提交互相冲突 | 每个 worktree 独立分支 wt/{name}，互不干扰 |
   | 并发安全 | 文件读写竞态，代码可能被覆盖 | 目录隔离，无竞态 |
   | 回滚 | 整个仓库回滚影响所有人 | 只影响一个 worktree（或直接丢弃） |

   worktree 的本质是 **git 提供的空间隔离**。`git worktree add` 创建已有仓库的独立 checkout——共享同一个 `.git` 目录（分支互见），但工作目录完全独立（文件互不见）。

2. **validate_worktree_name：安全的第一道防线**
   ```python
   VALID_WT_NAME = re.compile(r'^[A-Za-z0-9._-]{1,64}$')

   def validate_worktree_name(name):
       if not name: return error
       if name == "." or name == "..": return error  # 拒绝路径遍历
       if not VALID_WT_NAME.match(name): return error  # 仅允许安全字符
   ```
   为什么不能信任 LLM 传的 name 参数？因为如果 name 是 `"../../etc"`，`WORKTREES_DIR / name` 会逃逸出 `.worktrees/` 目录。正则 `[A-Za-z0-9._-]{1,64}` 使用白名单而非黑名单——只允许字母、数字、点、下划线、横线，1-64 字符长。`.` 和 `..` 虽然能匹配正则，但在函数开头被硬编码拒绝。

3. **create_worktree：校验 → 创建 → 绑定 → 记录**
   ```python
   def create_worktree(name, task_id=""):
       err = validate_worktree_name(name)   # ① 先校验
       if err: return f"Error: {err}"
       if path.exists(): return exists      # ② 防重复
       ok, result = run_git(["worktree", "add", ..., "-b", f"wt/{name}", "HEAD"])  # ③ git 操作
       if not ok: return f"Git error: {result}"
       if task_id: bind_task_to_worktree(task_id, name)  # ④ 可选绑定任务
       log_event("create", name, task_id)  # ⑤ 记录事件
   ```
   五步流程的每一步都有失败处理。`run_git` 返回 `(ok, output)` 元组而非抛异常——git 命令失败不应崩溃整个 Agent。分支命名 `wt/{name}` 把所有 worktree 分支放在 `wt/` 命名空间下，不会和正常开发分支（`main`、`feature/x`）混淆。

4. **remove_worktree 的安全检查：先数数，再决定**
   ```python
   def _count_worktree_changes(path):
       r1 = subprocess.run(["git", "status", "--porcelain"], cwd=path, ...)
       files = len([l for l in r1.stdout.splitlines() if l.strip()])
       r2 = subprocess.run(["git", "log", "@{push}..HEAD", "--oneline"], cwd=path, ...)
       commits = len([l for l in r2.stdout.splitlines() if l.strip()])
       return files, commits

   def remove_worktree(name, discard_changes=False):
       if not discard_changes:
           files, commits = _count_worktree_changes(path)
           if files > 0 or commits > 0:
               return f"Worktree '{name}' has {files} uncommitted file(s) "  # 拒绝
       # ... force remove
       run_git(["branch", "-D", f"wt/{name}"])  # 删除分支
   ```
   `remove_worktree` 默认拒绝删除有未提交变更的 worktree——防止 teammate 的代码被误删。`discard_changes=true` 是"我知道我在做什么"的显式确认。拒绝消息中还提示了备选方案：`keep_worktree` 保留供人工审查。

   注意顺序：先 `git worktree remove`，再 `git branch -D`。如果分支删除失败（比如是当前分支），worktree 目录还在，状态一致。分支删除失败不抛异常——`run_git` 返回 `(False, error_msg)` 但函数继续执行。

5. **wt_ctx：teammate 的 "当前目录指针"**
   ```python
   wt_ctx = {"path": None}              # 用 dict 实现闭包内可变引用

   def _wt_cwd():
       p = wt_ctx["path"]
       return Path(p) if p else None

   def _run_claim_task(task_id):
       result = claim_task(task_id, owner=name)
       if "Claimed" in result:
           task = load_task(task_id)
           if task.worktree:
               wt_ctx["path"] = str(WORKTREES_DIR / task.worktree)  # ← 设置 cwd
           else:
               wt_ctx["path"] = None                                  # ← 清除 cwd

   def _run_complete_task(task_id):
       result = complete_task(task_id)
       wt_ctx["path"] = None  # ← 完成时清除绑定
   ```
   为什么用 `dict` 而不是普通变量？因为 `wt_ctx` 需要在外层 `run()` 的闭包中被读取，同时在内层子函数（`_run_claim_task`、`_run_complete_task`）中被修改。Python 的闭包对不可变类型（`str`）的回写需要 `nonlocal` 声明；`dict` 的写只改内容不改引用，天然支持闭包内修改。这是 Python 闭包模式中的一种实用 idiom。

   cwd 的设置时机：**claim_task 时切换**，不是在 idle_poll 时设置。因为 claim 可能是 LLM 在 WORK 阶段调用的（不是自动认领），两个入口都需要更新 cwd。

6. **safe_path / run_bash / run_read / run_write 的 cwd 升级**
   ```python
   def safe_path(p, cwd=None):
       base = cwd or WORKDIR              # ← 可选 workdir
       path = (base / p).resolve()
       if not path.is_relative_to(base):  # ← 校验基于 cwd，不是 WORKDIR
           raise ValueError(...)

   def run_bash(command, cwd=None):
       r = subprocess.run(command, cwd=cwd or WORKDIR, ...)  # ← 执行目录可变
   ```
   这是 s17 → s18 最"静默"但最关键的基础设施升级。之前的 `safe_path` 固定基于 `WORKDIR` 做校验和拼接——不支持 worktree 子目录。s18 的 `cwd` 参数让所有文件操作可以透明切换到 worktree 目录：
   - `cwd=None` → 默认主仓库（与 s17 行为一致）
   - `cwd=.worktrees/auth` → 在 worktree 中操作（新增行为）

7. **events.jsonl：不可篡改的生命周期日志**
   ```python
   def log_event(event_type, worktree_name, task_id=""):
       event = {"type": event_type, "worktree": worktree_name,
                "task_id": task_id, "ts": time.time()}
       with open(events_file, "a") as f:
           f.write(json.dumps(event) + "\n")
   ```
   每行一条 JSON，追加写入。只写不读——教学版不需要实现日志查询。真实用途：出问题时可以 `cat events.jsonl` 查看所有 worktree 的创建/删除/保留记录，追溯"谁在什么时候创建了哪个 worktree 做什么任务"。

8. **s17 → s18：加了 1 个 dataclass 字段 + 6 个函数 + 3 个工具 + cwd 基础设施**
   | 新增 | 说明 |
   |------|------|
   | `Task.worktree` 字段 | 任务绑定 worktree 名称（`str \| None`） |
   | `validate_worktree_name()` | 正则白名单校验，拒绝路径遍历和非法字符 |
   | `run_git()` | 执行 git 命令，返回 `(ok, output)` 元组 |
   | `log_event()` | 追加生命周期事件到 events.jsonl |
   | `create_worktree()` | 创建 git worktree + 可选绑定任务 + 记录事件 |
   | `bind_task_to_worktree()` | 仅写 worktree 字段，保持 status=pending |
   | `remove_worktree()` | 安全检查后删除，discard_changes 跳过检查 |
   | `keep_worktree()` | 保留 worktree 供审查 + 记录事件 |
   | `_count_worktree_changes()` | 统计未提交文件数和未推送提交数 |
   | `safe_path` / `run_bash` / `run_read` / `run_write` + `cwd` | 所有文件/命令操作支持可选 workdir |
   | Teammate `wt_ctx` + `_wt_cwd` | 跟踪当前 worktree，claim/complete 时自动切换 |
   | 3 new Lead tools | create_worktree / remove_worktree / keep_worktree |

   s18 的核心变化不是"加了几个工具"，而是**把 Agent 的文件操作从"全局一个目录"升级为"每个任务独立目录"**。`cwd` 参数穿过 `safe_path` → `run_bash`/`run_read`/`run_write` 整个调用链，所有 I/O 操作透明地重定向到 worktree。
