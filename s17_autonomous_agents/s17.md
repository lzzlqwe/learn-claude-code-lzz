"不等人派活, 自己找活干" -- 空闲轮询 + 自动认领 + WORK/IDLE 双阶段生命周期。

Harness 层: 自主 -- Agent 空闲时扫任务板, 有活就抢, 没活就等, 超时就关。

```python
s17_autonomous_agents/code.py - 带自主任务认领的 Agent

在 s16 的 Team 协议之上，新增自主 Agent 能力。Teammate 不再是"等 Lead 派活"的被动角色，
而是主动扫描任务板（scan_unclaimed_tasks）→ 自动认领（auto-claim）→ 执行的自主工作者。
WORK/IDLE 双阶段生命周期让 Agent 在完成 LLM turn 后不死等协议消息，而是主动找下一个任务。

3 大自主机制:
  scan_unclaimed_tasks: 扫描 .tasks/ 目录 → 找 pending + 无 owner + 依赖完成的待办
  idle_poll: 5s 间隔轮询 60s → 收件箱/任务板有新任务 → 回到 WORK → 超时 → SHUTDOWN
  WORK/IDLE 双阶段: 外层 while True 驱动 WORK（10 轮 LLM）→ IDLE（轮询等待）循环

架构图：
    ┌──────────────────────────────────────────────────────────┐

    │  │    LLM turn (8 tools)               │                 │

         s17 的 teammate 是包工头——干完了自己去看任务板，有活就抢，
```

### 💡 核心设计模式解读：

1. **s16 被动等待 → s17 主动认领：多了什么？**
   | | s16 teammate | s17 teammate |
   |---|---|---|
   | 任务来源 | Lead 通过 send_message 派活 | 自己扫描任务板 + Lead 派活（两者都行） |
   | 空闲行为 | idle loop 等协议消息（被动） | idle_poll 扫收件箱 + 扫任务板（主动） |
   | 生命周期 | while not shutdown_requested（单循环） | WORK/IDLE 双阶段（外层 while True 驱动） |
   | 工具数量 | 5（bash/read/write/send/submit_plan） | 8（+ list_tasks/claim_task/complete_task） |
   | 身份保持 | 无 | Identity re-injection（上下文压缩后恢复记忆） |

   s16 的 teammate 需要 Lead 记住"我还派了谁干什么"——因为 teammate 不知道自己是谁。s17 的 teammate 有了任务板的自主能力：扫任务 → 认领 → 执行 → 完成 → 再扫下一个。Lead 只需要把任务写到板上，不用记得谁在干什么。

2. **scan_unclaimed_tasks：三个条件缺一不可**
   ```python
   def scan_unclaimed_tasks():
       for f in sorted(TASKS_DIR.glob("task_*.json")):
           task = json.loads(f.read_text())
           if (task.get("status") == "pending"       # ① 未开始
                   and not task.get("owner")          # ② 无人认领
                   and can_start(task["id"])):        # ③ 依赖已完成
               unclaimed.append(task)
   ```
   三个条件：
   - **status == pending**：只认领未开始的任务（已完成/进行中的跳过）
   - **not owner**：防止多个 teammate 抢同一个任务（谁先 claim 谁拿到）
   - **can_start**：依赖的前置任务必须全部 completed（被阻塞的任务不抢）

   注意：`scan_unclaimed_tasks` 只读文件不写文件——真正的"抢"发生在 `claim_task` 时（owner 检查 + status 更新是原子的文件写入）。

3. **idle_poll：12 轮 × 5s = 60s 的三通道轮询**
   ```python
   def idle_poll(agent_name, messages, name, role) -> str:
       for _ in range(12):           # 12 × 5s = 60s
           time.sleep(5)
           
           # 通道 1: 收件箱 → 协议消息
           inbox = BUS.read_inbox(agent_name)
           for msg in inbox:
               if msg.get("type") == "shutdown_request":
                   return "shutdown"  # ← 即使 IDLE 也能处理 shutdown
           if inbox:  # 有普通消息
               messages.append(...)
               return "work"          # ← 收件箱有活 → 回去干活
           
           # 通道 2: 任务板 → 自动认领
           unclaimed = scan_unclaimed_tasks()
           if unclaimed:
               result = claim_task(unclaimed[0]["id"], agent_name)
               if "Claimed" in result:
                   messages.append("<auto-claimed>...</auto-claimed>")
                   return "work"      # ← 抢到任务 → 回去干活
       
       # 通道 3: 超时
       return "timeout"               # ← 60s 无事 → 退出
   ```
   三个返回值的含义：
   - `"work"`：有新任务/消息 → 外层 while 回到 WORK 阶段
   - `"shutdown"`：收到关机协议 → break 外层 while → 发 summary → 退出
   - `"timeout"`：60s 无任何可做之事 → break 外层 while → 发 summary → 退出

   关键设计：shutdown 在 IDLE 阶段也能处理。s16 的 idle loop 用内层 while 处理协议，但需要在 WORK 阶段才能接收。s17 的 `idle_poll` 在每次 sleep 后都检查收件箱——shutdown 的最大延迟是 5s。

4. **claim_task 升级：owner 检查防并发争抢**
   ```python
   def claim_task(task_id, owner="agent"):
       task = load_task(task_id)
       if task.status != "pending":
           return f"Task {task_id} is {task.status}, cannot claim"
       if task.owner:                              # s17 新增！
           return f"Task {task_id} already owned by {task.owner}"
       if not can_start(task_id):
           # ... 返回阻塞详情
       task.owner = owner
       task.status = "in_progress"
       save_task(task)
   ```
   s12 的 `claim_task` 只检查 `status != "pending"`。s17 新增了 **owner 检查**——如果任务已经被其他 teammate 认领（owner 字段非空），直接拒绝。

   为什么需要？因为文件系统不是事务性的：两个 teammate 可能同时通过 `scan_unclaimed_tasks` 看到同一个任务（status=pending, owner=null），然后都去 claim。第一个 `claim_task` 写入 owner → 第二个再读文件时 owner 已经不为 null → 被拒绝。这是一种**乐观锁**——先抢后校验，无锁设计。

5. **Identity re-injection：上下文压缩后恢复记忆**
   ```python
   if len(messages) <= 3:    # 消息历史极少 → 可能被压缩了
       messages.insert(0, {"role": "user",
           "content": f"<identity>You are '{name}', role: {role}. "
                      f"Continue your work.</identity>"})
   ```
   teammate 长对话过程中，`messages[-20:]` 截断会丢掉最初的 SYSTEM prompt 信息。如果消息历史只剩 3 条以内（可能被 s08 的 compaction 清空），重新注入身份标识到 messages[0]。`Continue your work` 提示模型"你不是刚启动，是正在工作中"。

   真实 CC 的 identity re-injection 更精细——在 compaction 事件触发时注入，而不是简单检查消息数量。教学版用 `len(messages) <= 3` 做近似判断。

6. **teammate 工具从 5 到 8：任务板自治的工具基础**
   ```python
   sub_tools = [
       bash, read_file, write_file,       # 执行能力（s15 已有）
       send_message, submit_plan,         # 通信能力（s15/s16 已有）
       list_tasks, claim_task, complete_task,  # 任务自治（s17 新增）
   ]
   ```
   新增的三个工具让 teammate 拥有了完整的任务生命周期管理能力：
   - `list_tasks`：看任务板有什么活
   - `claim_task(owner=name)`：认领任务（owner 设为 teammate 名）
   - `complete_task`：完成当前任务

   但注意——这三个工具和 `idle_poll` 中的自动认领是**互补**而不是替代。自动认领导在 IDLE 阶段，工具调用发生在 WORK 阶段的 LLM turn 内。两者协作：IDLE 自动抢第一个任务 → WORK 阶段 LLM 调 `list_tasks` 检查有无更多任务 → 做完调 `complete_task` → 再调 `claim_task` 认领下一个。

7. **s17 agent_loop 简化：s17 不再有 cron + 后台任务**
   s17 的 `agent_loop` 回归到 s12 级别的简化版：
   ```python
   def agent_loop(messages, context):
       system = get_system_prompt(context)
       while True:
           response = client.messages.create(...)
           if response.stop_reason != "tool_use": return
           for block in response.content:
               handler = TOOL_HANDLERS.get(block.name)
               output = handler(**block.input)
               results.append({"type": "tool_result", ...})
           messages.append({"role": "user", "content": results})
   ```
   没有 `should_run_background`、没有 cron 队列、没有 queue_processor。教学版聚焦自主 Agent 的核心概念——复杂的 Harness 层功能可以独立展示，不互相干扰。

8. **s16 → s17：加了 2 个函数 + 3 个工具 + WORK/IDLE 生命周期**
   | 新增 | 说明 |
   |------|------|
   | `scan_unclaimed_tasks()` | 扫描任务板，找 pending + 无 owner + 依赖完成的待办 |
   | `idle_poll()` | 空闲轮询（收件箱 + 任务板），12 轮 × 5s，三通道返回 |
   | `IDLE_POLL_INTERVAL` / `IDLE_TIMEOUT` | 5s 间隔 / 60s 超时常量 |
   | Teammate `_run_list_tasks()` | teammate 的 list_tasks 闭包（显示任务板） |
   | Teammate `_run_claim_task()` | teammate 的 claim_task 闭包（owner=teammate 名） |
   | Teammate `_run_complete_task()` | teammate 的 complete_task 闭包 |
   | Identity re-injection | `len(messages) <= 3` 时重新注入身份 |
   | `claim_task` 升级 | 新增 owner 检查和详细阻塞报告 |
   | 3 new teammate tools | list_tasks / claim_task / complete_task（工具 5→8） |

   s17 的核心变化是**改变了 teammate 的生命周期模型**。s15 是 "干 10 轮就走"，s16 是 "有活就干、没活就等"，s17 是 "活干完了自己找下一个，找不到才关机"。这个递进反映了从"一次性助手"到"常驻工作者"的演化。
