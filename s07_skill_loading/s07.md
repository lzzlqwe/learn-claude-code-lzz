### 💡 核心设计模式解读：

1. **为什么要两级加载？全塞 SYSTEM prompt 会出什么问题？**
   ```
   SYSTEM = (
       f"You are a coding agent. "
       + open("react-style.md").read()     # 2000 行 → 每轮都带
       + open("sql-style.md").read()       # 1500 行 → 改 CSS 也要带
       + open("api-design.md").read()      # 3000 行 → 修 bug 也要带
   )  # 6500 行 SYSTEM prompt, 99% 内容与当前任务无关
   ```
   两级加载的核心思想：**区分配"知道有什么"和"知道具体内容"**。
   - Layer 1（目录）：Agent 每轮都知道有哪些技能可用，但只看到一行描述，token 开销极小
   - Layer 2（内容）：Agent 决定需要某个技能时，调 `load_skill` 加载完整文档。内容通过 tool_result 进入当前 messages，后续轮次随历史携带

2. **安全设计：为什么用 SKILL_REGISTRY 而不是直接读文件？**
   ```python
   # 启动时扫描，构建字典
   SKILL_REGISTRY[name] = {"name", "description", "content"}

   # 运行时只从字典取（不给 LLM 传文件路径的机会）
   def load_skill(name: str) -> str:
       skill = SKILL_REGISTRY.get(name)    # ← 字典查表，name 不是文件路径
       if not skill:
           return f"Skill not found: {name}"
       return skill["content"]
   ```
   LLM 传 `name="code-review"` — 查字典返回内容。LLM 传 `name="../../etc/passwd"` — 字典里没有这个 key，返回 `"Skill not found"`。**攻击者无法通过 name 参数读取任意文件，因为 name 从来就不是文件路径。**

3. **启动流程：从 skills/ 目录到 SYSTEM prompt**
   ```
   启动 → _scan_skills() → 遍历 skills/ 子目录 → 解析 SKILL.md frontmatter
         → 填充 SKILL_REGISTRY → list_skills() → build_system() → SYSTEM
   ```
   ```python
   def _scan_skills():
       for d in sorted(SKILLS_DIR.iterdir()):
           manifest = d / "SKILL.md"
           raw = manifest.read_text(encoding="utf-8")
           meta, body = _parse_frontmatter(raw)
           name = meta.get("name", d.name)
           desc = meta.get("description", raw.split("\n")[0].lstrip("#").strip())
           SKILL_REGISTRY[name] = {"name": name, "description": desc, "content": raw}

   _scan_skills()  # 启动时执行一次

   SYSTEM = build_system()  # 把目录注入 SYSTEM prompt
   ```
   整个扫描只在启动时跑一次。之后 SYSTEM prompt 的内容固定不变，不管 Agent 是否实际调用 `load_skill`。

4. **skill 内容注入后的流向**
   load_skill 返回的内容通过 tool_result 进入 messages：
   ```
   messages = [
       {"role": "user", "content": "review my code"},
       {"role": "assistant", "content": [tool_use: load_skill("code-review")]},
       {"role": "user", "content": [tool_result: "# Code Review Guidelines\n\n..."]},  ← 注入点
       {"role": "assistant", "content": [tool_use: read_file("app.py")]},
       {"role": "user", "content": [tool_result: "def main():\n    pass"]},
       {"role": "assistant", "content": [text: "根据 Code Review 规范，你的代码..."]},
   ]
   ```
   后续每次 LLM 调用都带着这份 skill 内容，直到它被上下文压缩（s08）或超出窗口被截断。

5. **s06 → s07：加了一个工具 + 启动时一段扫描**
   | 新增 | 改动量 | 说明 |
   |------|--------|------|
   | `_parse_frontmatter()` | +10 行 | 解析 YAML frontmatter |
   | `_scan_skills()` + `SKILL_REGISTRY` | +15 行 | 启动时扫描，构建注册表 |
   | `list_skills()` + `build_system()` | +12 行 | 生成动态 SYSTEM prompt |
   | `load_skill()` | +5 行 | 工具函数，查表返回内容 |

   循环不变，hook 不变，子 Agent 不变。所有 skill 逻辑完全在工具层完成。
