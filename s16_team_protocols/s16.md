"不只是聊天, 得有规矩" -- 请求-响应协议 + request_id 关联 + 状态机 + 空闲循环。

Harness 层: 协议 -- 结构化的 Agent 间通信, 状态可追踪, 响应可匹配。

```python
s16_team_protocols/code.py - 带 Team 协议系统的 Agent

在 s15 的 MessageBus + Teammate 之上，新增结构化协作协议。Lead 和 Teammate 之间
不再只发纯文本消息，而是走"请求 → 分发 → 处理 → 响应 → 匹配"的完整协议流程。
ProtocolState 状态机追踪每次协议交互，request_id 关联请求与响应。

3 大协议机制:
  ProtocolState + pending_requests: request_id 追踪 → status 状态机（pending→approved/rejected）
  match_response: consume_lead_inbox 中自动路由协议响应 → 类型校验 → 防重复处理
  Teammate idle loop: 子 LLM turn 结束后不退出，等待收件箱中的 shutdown/plan_approval 消息

架构图：
    ┌──────────────────────────────────────────────────────────┐
    │                     Lead Agent                           │
    │                                                          │
    │  request_shutdown("alice")                                │
    │    → new_request_id() → "req_123456"                     │
    │    → pending_requests["req_123456"] = ProtocolState(...)  │
    │    → BUS.send("shutdown_request", {request_id})          │
    │                                                          │
    │  request_plan("bob", "write tests")                      │
    │    → BUS.send("message", "Please submit a plan...")      │
    │                                                          │
    │  agent_loop 返回后:                                       │
    │    consume_lead_inbox()                                   │
    │      → BUS.read_inbox("lead")                             │
    │      → 遍历每条消息:                                       │
    │          type="shutdown_response"?                        │
    │            → match_response("shutdown_response", req_id)   │
    │              → 校验 type 匹配                              │
    │              → 校验 status == pending（防重复）            │
    │              → pending_requests[req_id].status = approved  │
    └──────────────────┬───────────────────────────────────────┘
                       │       MessageBus
                       ▼
    ┌──────────────────────────────────────────────────────────┐
    │              Teammate "alice" (后台线程)                   │
    │                                                          │
    │  while not shutdown_requested:        ← idle loop        │
    │    BUS.read_inbox("alice")                               │
    │      → 遍历收件箱消息:                                    │
    │          handle_inbox_message(msg):                      │
    │            type="shutdown_request"?                       │
    │              → BUS.send("shutdown_response", approve=True)│
    │              → return True (停止)                          │
    │            type="plan_approval_response"?                 │
    │              → approve? append "[Plan approved]"          │
    │              → reject?  append "[Plan rejected] + feedback│
    │              → return False (继续)                         │
    │                                                          │
    │    LLM turn (bash/read/write/send_message/submit_plan)    │
    │                                                          │
    │    stop_reason != tool_use?                               │
    │      → 进入 idle 等待 (sleep 1s + 检查收件箱)             │
    │      → 有协议消息 → dispatch → 回到 LLM turn              │
    │      → shutdown_request → 退出                            │
    │                                                          │
    │  BUS.send("result", summary) → Lead 收件箱                │
    └──────────────────────────────────────────────────────────┘

    协议交互时序（shutdown 协议）:
      Lead                        MessageBus              Teammate
      │                             │                      │
      │──shutdown_request(req_001)─→│                      │
      │  pending_requests[req_001]  │                      │
      │  = {status: "pending"}     │                      │
      │                             │──shutdown_request──→│
      │                             │                      │──handle: approve
      │                             │←─shutdown_response──│  + return True
      │←─consume_lead_inbox()──────│                      │  (停止线程)
      │  match_response(...)        │                      │
      │  → status = "approved" ✓   │                      │
      │                             │                      │

核心洞察："s15 是发微信——'帮我写个测试'，发了就发了。s16 是走流程——
         '请你关机'(request_id)→'收到, 马上关'(response, 同一个 request_id)
         → Lead 校验'确实是 shutdown 的回复, 不是别的'→ 标记完成。
         结构化、可追踪、防错配。"
```

### 💡 核心设计模式解读：

1. **s15 纯文本消息 → s16 结构化协议：加了什么？**
   | | s15 消息 | s16 协议 |
   |---|---|---|
   | 消息格式 | `{from, to, content, type}` | `{from, to, content, type, metadata: {request_id}}` |
   | 关联机制 | 无（纯文本，无法匹配请求-响应） | `request_id` 关联 + `match_response` 类型校验 |
   | 状态追踪 | 无 | `ProtocolState`（pending → approved/rejected） |
   | Type 语义 | `"message"` / `"result"` | `"shutdown_request"` / `"plan_approval_response"` 等 |
   | Teammate 生命周期 | 10 轮后强制退出 | idle loop（等待协议消息，shutdown 时优雅退出） |

   s15 的 teammate 是"干 10 轮就下班"——不管活干没干完。s16 的 teammate 是"活干完了就待命，收到 shutdown 才走"。这个变化的核心动力是协议系统：没有 shutdown_request，teammate 不知道何时该退出。

2. **match_response：不是简单的 dict.get，是三关校验**
   ```python
   def match_response(response_type, request_id, approve):
       state = pending_requests.get(request_id)
       if not state:
           return              # 第 1 关: request_id 必须存在于 pending_requests
       if state.type == "shutdown" and response_type != "shutdown_response":
           return              # 第 2 关: response_type 必须与请求类型匹配
       if state.type == "plan_approval" and response_type != "plan_approval_response":
           return              #      （防止 teammate 用错误类型的响应糊弄 Lead）
       if state.status != "pending":
           return              # 第 3 关: 状态必须是 pending（防重复处理）
       state.status = "approved" if approve else "rejected"
   ```
   三关校验防止：
   - **伪造 request_id**：不在 `pending_requests` 里 → 拒绝（第 1 关）
   - **类型不匹配**：Lead 发了 shutdown_request，teammate 回个 plan_approval_response → 拒绝（第 2 关）
   - **重复响应**：同一个 request_id 被回复两次 → 第二次被 `status != "pending"` 拦截（第 3 关）

3. **consume_lead_inbox：统一收件箱消费入口**
   ```python
   def consume_lead_inbox(route_protocol=True) -> list[dict]:
       msgs = BUS.read_inbox("lead")
       if route_protocol:
           for msg in msgs:
               meta = msg.get("metadata", {})
               req_id = meta.get("request_id", "")
               msg_type = msg.get("type", "")
               if req_id and msg_type.endswith("_response"):
                   match_response(msg_type, req_id, meta.get("approve", False))
       return msgs
   ```
   s15 的问题：`check_inbox` 工具和主循环各自调 `BUS.read_inbox("lead")`——但 `read_inbox` 是 destructive 的（读后即删）。如果 `check_inbox` 工具先读了收件箱，协议响应就被消费了，主循环再也看不到。

   s16 的解决方案：**统一入口**。`consume_lead_inbox` 是唯一调用 `BUS.read_inbox("lead")` 的地方，`check_inbox` 工具和主循环都经过它。协议路由（`match_response`）在读取后立即执行，不管谁调用都不会漏掉。

4. **Teammate idle loop：从 10 轮 exit 到无限等待**
   ```
   s15: for _ in range(10): LLM turn → exit
   s16: while not shutdown_requested:
           检查收件箱 → dispatch 协议消息
           LLM turn
           if stop_reason != tool_use:
               while not shutdown_requested:     ← idle 子循环
                   sleep(1)
                   检查收件箱 → 有协议消息? → dispatch
                   shutdown_request? → 退出 idle → 退出主循环
                   普通消息? → 注入 messages → break (回到 LLM turn)
   ```
   三层嵌套 while：
   - **外层 while**：主循环，shutdown_request 时退出
   - **内层 while**（idle）：LLM 无工具调用时进入，等待新消息唤醒
   - 如果收到普通消息（新任务），break 内层回到 LLM turn

   idle loop 的代价是一个 `sleep(1)` 的守护线程——CPU 开销几乎为零，但 teammate 可以跨分钟甚至跨小时待命。

5. **handle_inbox_message：teammate 端的协议分发器**
   ```python
   def handle_inbox_message(name, msg, messages) -> bool:
       msg_type = msg.get("type", "message")
       meta = msg.get("metadata", {})
       req_id = meta.get("request_id", "")

       if msg_type == "shutdown_request":
           BUS.send(name, "lead", "Shutting down gracefully.",
                    "shutdown_response",
                    {"request_id": req_id, "approve": True})
           return True   # ← 停止主循环

       if msg_type == "plan_approval_response":
           if approve:
               messages.append({"role": "user",
                   "content": "[Plan approved] Proceed with the task."})
           else:
               messages.append({"role": "user",
                   "content": f"[Plan rejected] Feedback: {msg['content']}"})
       return False  # ← 继续
   ```
   返回 `True` 表示"该停了"（目前只有 shutdown_request），返回 `False` 表示"继续干活"。新增协议类型只需在 if 链中添加分支——开闭原则的简化实现。

6. **submit_plan 协议：teammate 主动发起审批**
   ```python
   def _teammate_submit_plan(from_name, plan):
       req_id = new_request_id()
       pending_requests[req_id] = ProtocolState(
           type="plan_approval", sender=from_name,
           target="lead", status="pending", payload=plan)
       BUS.send(from_name, "lead", plan, "plan_approval_request",
                {"request_id": req_id})
       return f"Plan submitted ({req_id}). Waiting for approval..."
   ```
   和 shutdown 不同，submit_plan 是 teammate 主动发起的请求。执行流程：
   1. Teammate 调 `submit_plan("我将执行以下步骤...")`
   2. `_teammate_submit_plan` 创建 ProtocolState → BUS.send 给 Lead
   3. **Teammate 线程继续运行**（不阻塞等待审批）
   4. Lead 的 `consume_lead_inbox` 收到 `plan_approval_request`
   5. Lead 调 `review_plan(req_id, approve=True/False, feedback="...")` 
   6. BUS.send `plan_approval_response` → Teammate 的 `handle_inbox_message` 收到审批结果
   7. Teammate 注入 `[Plan approved]` 或 `[Plan rejected]` 到自己的 messages

   注意：这是**协议级审批**，不是代码级门禁。提交后 teammate 线程继续运行——它仍然可以调 bash/write。真正的工具拦截需要阻塞 teammate 的工具分发直到审批到达，教学版未实现此层。

7. **review_plan：Lead 端的审批决策**
   ```python
   def run_review_plan(request_id, approve, feedback=""):
       state = pending_requests.get(request_id)
       if not state: return f"Request {request_id} not found"
       if state.status != "pending": return f"already {state.status}"
       state.status = "approved" if approve else "rejected"
       BUS.send("lead", state.sender, feedback or "...",
                "plan_approval_response",
                {"request_id": request_id, "approve": approve})
   ```
   Lead 的 `review_plan` 做了三件事：状态校验（防重复审批）→ 更新本地状态 → 通过 BUS 发送审批结果。`request_id` 是贯穿始终的关联键——teammate 提交时生成、Lead 审批时引用、teammate 收响应时匹配。

8. **s15 → s16：加了 1 个 dataclass + 5 个函数 + 3 个工具 + idle loop**
   | 新增 | 说明 |
   |------|------|
   | `ProtocolState` dataclass | 协议状态数据类（6 字段 + 状态机） |
   | `pending_requests` dict | 进行中的协议请求追踪（request_id → ProtocolState） |
   | `new_request_id()` | 生成唯一请求 ID（req_{random}） |
   | `match_response()` | 响应-请求关联 + 三关校验（存在性/类型匹配/防重复） |
   | `consume_lead_inbox()` | 统一收件箱消费入口（check_inbox + 主循环共用） |
   | `handle_inbox_message()` | Teammate 端协议消息分发器（shutdown/plan_approval） |
   | `_teammate_submit_plan()` | Teammate 提交计划审批 |
   | `run_request_shutdown()` | Lead 发送 shutdown 协议请求 |
   | `run_request_plan()` | Lead 要求 teammate 提交计划 |
   | `run_review_plan()` | Lead 审批/拒绝 teammate 的计划 |
   | Teammate idle loop | 从 `for _ in range(10)` 升级为 `while not shutdown_requested` |
   | Bus.send 新增 `metadata` 参数 | 携带 `request_id` / `approve` 等协议元数据 |
   | `schedule_cron` / `list_crons` / `cancel_cron` 移除 | s16 不再有 cron 调度器（简化教学） |

   s16 的核心变化不是"加了几个工具"，而是**把 s15 的纯文本消息通信升级为结构化协议通信**。request_id 是协议层的"TCP 序列号"，ProtocolState 是"连接状态表"，match_response 是"校验和"。
