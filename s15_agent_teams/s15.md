"一个人干不完, 组队干" -- 文件邮箱通信, 后台线程跑 teammate, Lead 收件箱注入结果。

Harness 层: 协作 -- 多 Agent 并发执行, 消息总线解耦, 共享文件系统。

### 💡 核心设计模式解读：

1. **Lead-Teammate 架构：不是"调用子函数"，是"雇佣外包"**
   | | 子 Agent (s06) | Teammate (s15) |
   |---|---|---|
   | 启动方式 | 同步 spawn → 等结果 | 后台线程 spawn → 立即返回 |
   | 通信 | 一次性 prompt → summary 返回 | 双向 MessageBus（可多次收发） |
   | 生命周期 | 一次性，返回后销毁 | 后台持续运行，最多 10 轮 |
   | 并发 | 串行（等一个完再下一个） | 并行（多个 teammate 同时跑） |
   | 上下文 | 独立 messages[]，隔离 | 独立 messages[]，隔离 + 收件箱注入 |

   s06 的 subagent 是"你来帮我算一下，算完告诉我"——同步调用，结果直接返回。s15 的 teammate 是"你去那边干活，有问题给我写信"——后台线程独立运行，通过 MessageBus 双向通信。Lead spawn 后立即继续自己的工作，不等 teammate 完成。

2. **MessageBus：文件即邮箱，追加即发送，读取即消费**
   ```python
   class MessageBus:
       def send(self, from_agent, to_agent, content, msg_type="message"):
           inbox = MAILBOX_DIR / f"{to_agent}.jsonl"
           with open(inbox, "a") as f:
               f.write(json.dumps(msg) + "\n")    # 追加一行 JSON

       def read_inbox(self, agent) -> list[dict]:
           inbox = MAILBOX_DIR / f"{agent}.jsonl"
           if not inbox.exists():
               return []
           msgs = [json.loads(line) for line in inbox.read_text().splitlines()]
           inbox.unlink()  # ← 读取后删除，消费即销毁
           return msgs
   ```
   三个设计要点：
   - **文件即邮箱**：`to_agent.jsonl` 就是收件人的邮箱文件，每个 Agent 一个。不需要注册中心、不需要连接管理
   - **追加即发送**：`open("a")` 追加一行 JSON → 发送完成。写入顺序天然是时间顺序
   - **读取即消费**：`read_text()` + `unlink()` → 取走就没了。不重复投递，不积压

   教学版无文件锁——多个 Agent 同时往同一个文件追加，可能行交错。真实 CC 使用 `proper-lockfile` 保证原子写入。

3. **teammate 的 "受限代理循环"：4 个工具 + 10 轮限制 + 结果回传**
   ```python
   sub_tools = [bash, read_file, write_file, send_message]  # 只有 4 个工具
   sub_handlers = {
       "bash": run_bash, "read_file": run_read, "write_file": run_write,
       "send_message": lambda to, content: (BUS.send(name, to, content), "Sent")[1],
   }

   for _ in range(10):                    # 教学版 10 轮限制
       inbox = BUS.read_inbox(name)        # 每轮检查自己的收件箱
       if inbox:
           messages.append({"role": "user", "content": f"<inbox>{json.dumps(inbox)}</inbox>"})
       response = client.messages.create(model=MODEL, system=system,
                                         messages=messages[-20:],  # ← 只保留最近 20 条
                                         tools=sub_tools, max_tokens=8000)
       # ... 执行工具 ...
   ```
   teammate 不能 spawn_teammate（防止无限递归创建子 Agent）、不能 schedule_cron（只干活不管调度）、不能 claim_task（任务由 Lead 分配）。`send_message` 是它和 Lead 的唯一通信方式。

   `messages[-20:]` 切片控制 teammate 上下文窗口，防止长对话撑爆 token 限制。真实的 CC 用 s08 的 compaction 策略，教学版直接截断。

4. **teammate 的最终回传：从消息历史中提取 summary**
   ```python
   summary = "Done."
   for msg in reversed(messages):          # 从后往前找
       if msg["role"] == "assistant" and isinstance(msg["content"], list):
           for b in msg["content"]:
               if getattr(b, "type", None) == "text":
                   summary = b.text        # 取最后一条 assistant text
                   break
   BUS.send(name, "lead", summary, "result")
   ```
   teammate 完成后**从自己的消息历史中提取最后一条文本回复**作为结果摘要，通过 MessageBus 发给 Lead。不做 LLM 总结（省一次 API 调用），直接取原文。如果整个对话中没有任何 assistant text（全工具调用），返回默认 "Done."。

5. **Lead 的收件箱注入：主循环结束后统一收取**
   ```python
   agent_loop(history, context)          # 正常的 Agent 轮次
   # agent_loop 返回后（stop_reason != tool_use）
   inbox = BUS.read_inbox("lead")         # 收取所有 teammate 结果
   if inbox:
       inbox_text = "\n".join(
           f"From {m['from']}: {m['content'][:200]}" for m in inbox)
       history.append({"role": "user",
                       "content": f"[Inbox]\n{inbox_text}"})
       print(f"\n\033[33m[Inbox: {len(inbox)} messages injected]\033[0m")
   ```
   收件箱注入时机：**agent_loop 返回后**。如果 LLM 回复了 text（不调用工具），主循环结束 → 检查收件箱 → 注入。下轮用户输入时，LLM 能看到 `[Inbox] From alice: Done.`。

   这和 `check_inbox` 工具有什么区别？`check_inbox` 是 LLM 主动调用（"让我看看有没有 teammate 回信"），主循环注入是**自动的**——不管 LLM 有没有检查，下一轮开始前都会自动插入。两者互补：一个是被动推送，一个是主动拉取。

6. **active_teammates 字典：防止重名冲突**
   ```python
   active_teammates: dict[str, bool] = {}

   if name in active_teammates:
       return f"Teammate '{name}' already exists"

   active_teammates[name] = True
   # ... 线程运行 ...
   active_teammates.pop(name, None)  # 完成时移除
   ```
   一个简单的字典防止 Lead 重复创建同名 teammate。名字同时被用作邮箱文件名（`{name}.jsonl`）——如果两个 teammate 同名，它们会共享同一个收件箱，消息混乱。`active_teammates` 在 spawn 时注册、完成时移除，是一个轻量级并发控制。

7. **s14 → s15：加了 1 个类 + 4 个函数 + 3 个工具 + 1 个收件箱注入**
   | 新增 | 说明 |
   |------|------|
   | `MessageBus` class | 文件邮箱消息总线（send + read_inbox） |
   | `BUS` 全局实例 | 单例 MessageBus，所有 Agent 共用 |
   | `active_teammates` dict | 防重名 + 生命周期追踪 |
   | `spawn_teammate_thread` | 后台线程创建 teammate（独立 agent_loop） |
   | `run_spawn_teammate` | spawn_teammate 工具处理函数 |
   | `run_send_message` | send_message 工具处理函数（Lead → teammate） |
   | `run_check_inbox` | check_inbox 工具处理函数（Lead 主动查收） |
   | `MAILBOX_DIR` | .mailboxes/ 目录 |
   | Lead 收件箱注入逻辑 | agent_loop 返回后 `BUS.read_inbox("lead")` |

   s15 在 s14 的 Cron 调度器基础上新增了多 Agent 协作层。cron、后台任务、task system 全部保持不变。teammate 线程和 cron_scheduler_loop、queue_processor_loop 并列运行——三种不同的守护线程在同一进程中互不干扰。
