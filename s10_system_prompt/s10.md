### 💡 核心设计模式解读：

1. **为什么要把 prompt 拆成 section？**
   s01-s09 的 SYSTEM prompt 是一整段硬编码字符串：
   ```python
   SYSTEM = f"You are a coding agent at {WORKDIR}. Use tools to solve tasks. Act, don't explain. ..."
   ```
   三个问题：
   - **换项目要重写整个 prompt**，不知道哪些该改、哪些该留
   - **改一处可能影响全局**，加一段工具描述可能跟前面的指令冲突
   - **每次请求都带全部内容**，即使当前对话用不到某些段落也浪费 token

   拆成 section 后，每个 section 职责明确，独立维护。`tools` 改一行动不了 `identity`，`memory` 加一段不碰 `workspace`。

2. **按需加载：基于真实状态，不是关键词猜测**
   ```python
   def assemble_system_prompt(context: dict) -> str:
       sections = []
       # 始终加载
       sections.append(PROMPT_SECTIONS["identity"])
       sections.append(PROMPT_SECTIONS["tools"])
       sections.append(PROMPT_SECTIONS["workspace"])
       # 按需加载 — 基于真实状态
       memories = context.get("memories", "")
       if memories:
           sections.append(f"Relevant memories:\n{memories}")
       return "\n\n".join(sections)
   ```
   `if memories:` 判断的是 `context["memories"]` 是否为空——这是真实的文件系统状态（`MEMORY_INDEX.exists()` 的结果），不是在消息里搜关键词。如果消息里出现了 "memory" 这个词但 `.memory/` 目录是空的，memory section 不会加载。

3. **缓存：json.dumps 做确定性序列化**
   ```python
   def get_system_prompt(context: dict) -> str:
       key = json.dumps(context, sort_keys=True, ensure_ascii=False, default=str)
       if key == _last_context_key and _last_prompt:
           print("[cache hit]")
           return _last_prompt
       _last_prompt = assemble_system_prompt(context)
       return _last_prompt
   ```
   为什么用 `json.dumps` 而不是 Python 内置 `hash()`？
   - `hash()` 有进程随机化（PYTHONHASHSEED），不同进程相同数据可能得不同 hash
   - `hash()` 遇到 list/dict 直接报 `unhashable type`
   - `json.dumps(context, sort_keys=True)` 把字典变成确定性的字符串，同数据一定同 key

   注意：这里的缓存只是"避免重复拼接字符串"，和 Claude Code 的 API prompt cache 不是一回事。CC 用 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 把 prompt 分成不同 cache scope 的块，静态部分命中 global cache，这才真正节省 API 调用的 token 计费。

4. **context 是当前运行态的快照**
   ```python
   def update_context(context, messages):
       memories = ""
       if MEMORY_INDEX.exists():
           content = MEMORY_INDEX.read_text().strip()
           if content:
               memories = content
       return {
           "enabled_tools": list(TOOL_HANDLERS.keys()),   # 实际注册的工具
           "workspace": str(WORKDIR),                      # 当前工作目录
           "memories": memories,                           # 记忆索引内容（或空）
       }
   ```
   context 反映的是"当前 Agent 的真实状态"，不是"对话在聊什么"：
   - `enabled_tools` → 哪些工具注册了（不是消息里提到哪些）
   - `memories` → `.memory/` 目录下有文件吗（不是消息里说了"记住"）

5. **assemble vs get 的分工**
   ```
   assemble_system_prompt() → 只管拼接（纯函数，无副作用）
   get_system_prompt()       → 包裹一层缓存（有状态，但对外透明）
   ```
   agent_loop 只调 `get_system_prompt()`，不关心底层有没有缓存命中、有没有重新拼接。这是典型的**装饰器模式**——给纯函数包一层缓存，调用方无感。

6. **s09 → s10：加了一个字典 + 三个函数，循环改一行**
   | 新增 | 说明 |
   |------|------|
   | `PROMPT_SECTIONS` | 4 个 section 字典，按主题存放提示词片段 |
   | `assemble_system_prompt(context)` | 根据真实状态拼接 section |
   | `get_system_prompt(context)` | json.dumps 缓存 + 组装 |
   | `update_context(context, messages)` | 扫描文件系统更新 context 快照 |

   循环改动：把 `system = build_system()` 换成 `system = get_system_prompt(context)`，然后在每轮循环末尾 `context = update_context(context, messages)`。
