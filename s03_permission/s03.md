Harness 层: 权限 -- 在工具执行前加一道门。

```python
s03_permission/code.py - 权限控制系统

与 s02 相比，Agent 的核心交互循环只加了一行代码（check_permission）。
我们给每个工具调用装了三道闸门：硬黑名单 → 规则匹配 → 用户审批。

架构图：
    +----------+      +-------+      +---------------------+      +------------------+
    |   用户    | ---> |  LLM  | ---> |   三道闸门权限管道     | ---> |   工具执行        |
    |   提示词  |      |       |      | Gate 1: 硬黑名单      |      | bash/read/write  |
    +----------+      +---+---+      |   ↓ (命中→阻断)       |      | /edit/glob       |
                          ^          | Gate 2: 规则匹配      |      +------------------+
                          |          |   ↓ (命中→问用户)     |
                          +----------+ Gate 3: 用户审批      |
                          返回工具执行结果|   ↓ (y/N?)          |
                                     +---------------------+

核心洞察："agent_loop 只加了一行 if not check_permission(block): continue"
```

### 💡 核心设计模式解读：

1. **为什么要分三道闸门，而不是一道？**
   s01/s02 把危险命令黑名单嵌在 `run_bash` 函数内部。问题是：
   - **职责混乱**：工具函数既要"执行"，又要"判断安不安全"
   - **不可扩展**：给 write_file 加权限？得改 run_write。加用户审批？得改所有工具函数
   
   s03 把权限判断从工具函数中**抽离**出来，变成独立的三道闸门：
   - Gate 1（硬黑名单）：无条件封杀，比如 `rm -rf /`、`sudo`。**不问用户，直接拒绝。**
   - Gate 2（规则匹配）："这个操作有风险，但具体要不要做，交给用户决定"。比如 `rm somefile.txt`（可能是正常的清理）、写到工作目录外。
   - Gate 3（用户审批）：暂停等待用户输入 y/N。**默认拒绝**（直接回车 = no）。
   
   **三道都没命中 → 直接放行**。大部分日常操作（读文件、列目录、在工作目录内写文件）走这条路，用户无感知。

2. **规则匹配的扩展性**：
   观察 `PERMISSION_RULES` 的设计——它是一个**规则列表**，每条规则是 `{tools, check, message}` 三元组：
   ```python
   PERMISSION_RULES = [
       {"tools": ["write_file", "edit_file"],
        "check": lambda args: 是否写到工作目录外,
        "message": "Writing outside workspace"},
       {"tools": ["bash"],
        "check": lambda args: 命令是否包含 rm/chmod 777 等,
        "message": "Potentially destructive command"},
   ]
   ```
   以后想加新规则（比如"禁止删除 .env 文件"），只需往列表里追加一条，**不改任何函数**。这和 s02 的 TOOL_HANDLERS 查表模式是同一个思想——策略与机制分离。

3. **s02 → s03 的职责迁移**：
   对比 s02 的 `run_bash`：
   ```python
   # s02: 黑名单嵌在工具函数内部
   def run_bash(command):
       if any(d in command for d in ["rm -rf /", "sudo", ...]):
           return "Error: Dangerous command blocked"
       # ... 执行命令
   
   # s03: 黑名单提到 Gate 1，run_bash 只管执行
   def run_bash(command):
       # ... 只执行命令，不做权限判断
   
   def check_deny_list(command):  # 权限判断独立
       for pattern in DENY_LIST:
           if pattern in command:
               return f"Blocked: ..."
   ```
   **工具只负责"能不能做"（技术上），权限管道只负责"让不让做"（策略上）。** 这是生产级 Agent 的基础架构模式。

4. **安全边界再强调**：
   s02 的 `safe_path` 锁住了文件操作的工作目录。s03 在此基础上加了：
   - bash 的黑名单（Gate 1）：防止 `rm -rf /`、`mkfs` 等毁灭性命令
   - bash 的规则匹配（Gate 2）：`rm`、`chmod 777`、`> /etc/` 等需用户确认
   - write_file/edit_file 的规则匹配（Gate 2）：二次确认不写到工作目录外
   
   **教学提示**：Gate 1 的字符串匹配不是真正的安全机制（命令变体、shell 展开可以绕过）。生产环境（如 Claude Code）用的是多层规则 + 分类器 LLM + 权限冒泡机制。这里的教学版是刻意简化的。
