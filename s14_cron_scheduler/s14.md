Harness 层: 定时 -- 时间驱动的 Agent 唤醒, 持久化跨会话保留。

```python
s14_cron_scheduler/code.py - 带 Cron 定时调度器的 Agent
```

### 💡 核心设计模式解读：

1. **4 层解耦架构：各司其职，互不阻塞**
   | 层 | 组件 | 职责 | 运行方式 |
   |---|---|---|---|
   | Layer 1 | `cron_scheduler_loop` | 时间判断 + 表达式匹配 | 守护线程，每 1s 轮询 |
   | Layer 2 | `cron_queue` | 线程安全的任务缓冲 | `list[CronJob]` + Lock |
   | Layer 3 | `queue_processor_loop` | 检测队列 + 唤酟 Agent | 守护线程，每 0.2s 检查 |
   | Layer 4 | `agent_loop` | 消费任务 + 执行 LLM 调用 | 主线程/用户触发 |

   关键设计：调度器不直接调用 agent_loop，而是通过队列 + 队列处理器解耦。调度器只管 "到时间了没"，队列处理器只管 "Agent 有空没"。两者完全独立，互不知晓对方的存在。

2. **cron 表达式匹配：DOM/DOW 的 OR 语义**
   ```python
   # Minute, hour, month must all match
   if not (m and h and month_ok):
       return False
   # DOM and DOW: if both constrained, either matching is enough (OR)
   dom_unconstrained = dom == "*"
   dow_unconstrained = dow == "*"
   if dom_unconstrained and dow_unconstrained:
       return True       # * * → 每天都匹配
   if dom_unconstrained:
       return dow_ok     # 只有 DOW 约束 → 按星期匹配
   if dow_unconstrained:
       return dom_ok     # 只有 DOM 约束 → 按日期匹配
   return dom_ok or dow_ok  # 两者都约束 → OR
   ```
   标准 cron 的 DOM（月内第几天）和 DOW（周几）是 OR 关系。`"0 9 13 * 5"` 不要求既是 13 号又是周五，而是 13 号 **或** 周五的 9:00 都触发。这是 cron 语义里最容易踩坑的点——很多人以为它是 AND。

3. **防重复触发：日期感知的 minute_marker**
   ```python
   minute_marker = now.strftime("%Y-%m-%d %H:%M")  # "2026-06-05 09:00"
   if _last_fired.get(job.id) != minute_marker:
       cron_queue.append(job)
       _last_fired[job.id] = minute_marker
   ```
   `_last_fired` 记录每个任务最后一次触发的分钟标记。每小时级及以上频率的任务，在这一分钟内只会触发一次。`%Y-%m-%d %H:%M` 格式确保跨小时、跨天时 marker 不同，不会错误阻止下次触发。

4. **durable vs session：两种生命周期**
   | | durable=True | durable=False |
   |---|---|---|
   | 存储位置 | `.scheduled_tasks.json` | 仅内存 `scheduled_jobs` |
   | 跨会话保留 | ✓（重启后 `load_durable_jobs()` 恢复） | ✗（进程退出即消失） |
   | 持久化时机 | `schedule_job` / `cancel_job` 时写入 | 无 |
   | 典型用途 | "每天早上 9 点日报" | "5 分钟后提醒我" |

   `load_durable_jobs()` 在启动时执行，跳过 cron 表达式不合法的任务（防止磁盘文件被篡改导致启动崩溃）。`save_durable_jobs()` 每次增删 durable 任务时全量覆写文件，教学版简化实现。

5. **queue_processor_loop：不打断用户的 "小秘书"**
   ```python
   while True:
       time.sleep(0.2)
       if not has_cron_queue():
           continue                            # 队列空 → 继续睡觉
       if not agent_lock.acquire(blocking=False):
           continue                            # Agent 正忙 → 不打断
       try:
           if not has_cron_queue():            # 双重检查
               continue
           run_agent_turn_locked()             # 持锁执行 Agent turn
       finally:
           agent_lock.release()
   ```
   三个关键设计：
   - **0.2s 轮询**：比调度器的 1s 更快，确保定时任务到达后尽快投递
   - **非阻塞锁**：`acquire(blocking=False)` → Agent 正忙时直接跳过，不打搅
   - **双重检查**：获取锁后再次 `has_cron_queue()`，防止在等待锁期间队列被另一个线程清空

6. **agent_lock 互斥锁：防止用户输入和 cron 队列并发冲突**
   ```python
   # 用户交互入口
   with agent_lock:
       run_agent_turn_locked(query)

   # 队列处理器
   if not agent_lock.acquire(blocking=False):
       continue  # 用户在操作 → 跳过
   ```
   `agent_lock` 确保同一时间只有一个 Agent turn 在运行。用户在输入命令时，cron 队列处理器不会插队；cron 任务执行时，用户输入也会被阻塞。两个入口（用户 + 定时）共享同一条 `session_history`，不加锁会导致消息交错、LLM 看到混乱的上下文。

7. **cron 表达式校验：5 字段 × 各自范围 + 模式支持**
   ```python
   bounds = [(0, 59), (0, 23), (1, 31), (1, 12), (0, 6)]
   names = ["minute", "hour", "day-of-month", "month", "day-of-week"]
   ```
   `validate_cron()` 在 `schedule_job()` 注册时调用，拒绝非法表达式。`_validate_cron_field()` 递归校验四种模式：`*`（通配）、`*/N`（步进，N 必须 > 0）、`A,B,C`（枚举）、`A-B`（范围，起点 ≤ 终点 + 都在界内）。校验发生在注册时而非触发时，错误早发现。

   `schedule_job()` 返回值类型 `CronJob | str`：合法 → 返回 CronJob 对象，非法 → 返回错误字符串。LLM 通过 `run_schedule_cron` 的返回文本 "Error: ..." 判断注册是否成功。
