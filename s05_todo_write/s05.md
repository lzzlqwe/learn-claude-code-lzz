"没有计划的 agent 走哪算哪" -- 先列步骤再动手，长任务更不容易漏项。

Harness 层: 规划 -- 让 Agent 在动手之前先想清楚。

```python
s05_todo_write/code.py - 带有 Todo 列表和 Nag 提醒的 Agent

在 s04 的 hook 系统之上，新增一个 todo_write 工具。它本身不做任何实际工作——不能读文件、
不能跑命令——只是让 Agent 在动手前先分解任务、执行中更新状态。

    |   用户    | ---> |  LLM  | ---> |   工具分发                  |
    |   提示词  |      |       |      |   bash/read/write/edit/glob |
    +----------+      +---+---+      |   todo_write ← 新增(纯计划)   |
                          ^          +-------------+--------------+
                          |                        |
                          +------------------------+ 工具执行结果
                                        |
                            +-----------+-----------+
                            | CURRENT_TODOS (内存)   |
                            | [ ] 爬取数据             |
                            | [▸] 清洗数据 ← 正在做     |
                            | [✓] 设计API            |
                            +-----------------------+
                                        |
                        如果 rounds_since_todo >= 3:
                          注入 <reminder>Update your todos.

核心洞察："todo_write 不给 Agent 增加任何执行能力。它增加的是规划能力。"
```

### 💡 核心设计模式解读：

1. **todo_write：纯规划工具，零执行能力**
   ```python
   CURRENT_TODOS: list[dict] = []   # 全局内存变量，退出即清空

   def run_todo_write(todos: list) -> str:
       # 只做两件事: 1. 校验字段  2. 存入内存 + 打印彩色列表
       for i, t in enumerate(todos):
           if "content" not in t or "status" not in t:
               return f"Error: todos[{i}] missing 'content' or 'status'"
           if t["status"] not in ("pending", "in_progress", "completed"):
               return f"Error: todos[{i}] has invalid status ..."
       CURRENT_TODOS = todos
       # 终端输出彩色任务列表: pending=无图标, in_progress=青色▸, completed=绿色✓
       ...
   ```
   注意：教学版**没有**"同一时间只能有一个 in_progress"的限制。这个限制是 Claude Code 生产版（Task System V2 / TodoManager）才有的。教学版只做最基本的字段校验，保持概念量可控。

2. **Nag 提醒：外部监督者，不侵入工具逻辑**
   ```python
   rounds_since_todo = 0   # 模块级计数器

   def agent_loop(messages: list):
       global rounds_since_todo
       while True:
           # 连续 3 轮未更新 todo → 注入一条"用户消息"催促
           if rounds_since_todo >= 3 and messages:
               messages.append({"role": "user",
                   "content": "<reminder>Update your todos.</reminder>"})
               rounds_since_todo = 0

           # ... LLM 调用 ...

           rounds_since_todo += 1   # 每轮 +1
           for block in response.content:
               # ...
               if block.name == "todo_write":
                   rounds_since_todo = 0   # 调了 todo 就重置
   ```
   Nag 提醒是一种**非破坏性纠偏**。它不是报错、不是拒绝执行，而是在消息流里塞一条 `<reminder>`，让 LLM 自己看到后主动更新进度。这在 Agent 工程学中叫"动态 Prompt 注入"——不需要改模型参数，不需要中断循环。

3. **为什么需要 todo_write（而不只是 SYSTEM prompt 里说"请先计划"）？**
   仅靠 SYSTEM prompt 有两个问题：
   - **上下文稀释**：对话越长，工具结果不断填满上下文，SYSTEM prompt 的影响力逐渐衰减。Agent 做了十步之后可能已经"忘记"了要计划。
   - **用户不可见**：LLM 在脑子里计划的那些步骤，用户看不到。用了 `todo_write` 后，计划变成终端可见的彩色列表，用户随时知道 Agent 在干什么。

   `todo_write` 等于是给大模型外挂了一个**工作记忆区（Working Memory）**——它强制 Agent 把计划写到"纸"上，每次更新状态都能看到整体进度，不会迷失在局部细节中。

4. **SYSTEM prompt 的双重引导**
   ```python
   SYSTEM = (
       f"You are a coding agent at {WORKDIR}. "
       "Before starting any multi-step task, use todo_write to plan your steps. "
       "Update status as you go."
   )
   ```
   SYSTEM 说"before starting"，但 LLM 未必听。Nag 提醒说"你已经 3 轮没更新了"，是强制执行。两层引导互补：SYSTEM = 软引导，Nag = 硬提醒。

5. **s04 → s05：循环只多了计数器，Dispatch Map 完全没变**
   | 新增 | 改动量 | 改动位置 |
   |------|--------|---------|
   | `todo_write` 工具 | +30 行 | 独立函数 + schema + 一行映射 |
   | Nag 提醒 | +8 行 | agent_loop 顶部 + todo 检测重置 |
   | SYSTEM prompt | 改 1 行 | 加入 plan 引导 |
   | TOOL_HANDLERS | 加 1 行 | `"todo_write": run_todo_write` |

   循环结构没变，hook 系统没变，分发机制没变。新工具自动被 dispatch，不需要写任何 if/elif。
