"大目标拆成小任务, 排好序, 持久化" -- 文件持久化的任务图, 多 Agent 协作的基础。

Harness 层: 任务 -- 持久化的目标, 可恢复的进度。

```python
s12_task_system/code.py - 带文件持久化任务图的 Agent

在 s10 的 prompt 组装之上，新增 Task System。每个任务是一个 .tasks/{id}.json 文件，
支持 blockedBy 依赖关系，跨会话持久化。5 个工具覆盖完整任务生命周期。

架构图：
    ┌──────────────────────────────────────────────┐
    │              .tasks/ 目录（JSON 持久化）       │
    │  task_xxx.json: {subject, status, blockedBy}  │
    │  task_yyy.json: {subject, status, owner}      │
    └───────────────────┬──────────────────────────┘
                        │
    ┌───────────────────▼──────────────────────────┐
    │  create_task(subject, blockedBy, description) │  ← 创建 + 自动 save
    │  list_tasks() → 状态图标 + blockedBy          │
    │  get_task(id) → 完整 JSON                     │
    │  claim_task(id) → can_start 检查依赖          │  ← 依赖未完成 → 拒绝认领
    │  complete_task(id) → 报告 unblocked 下游任务    │
    └───────────────────┬──────────────────────────┘
                        │
    ┌───────────────────▼──────────────────────────┐
    │  can_start(task_id):                         │
    │    for dep in task.blockedBy:                │
    │      dep.json 不存在? → False (阻塞)           │
    │      dep.status != "completed"? → False       │
    │    → True (所有依赖已完成，可以开始)            │
    └──────────────────────────────────────────────┘

    Task 状态机: pending → claim → in_progress → complete → completed
                       ↑ 需要所有 blockedBy = completed ↑

核心洞察："s05 的 TodoWrite 是便利贴——进程挂了就没了。s12 的 Task System 是
         合同——每个任务落盘成 JSON，跨会话还在，依赖关系自动检查。"
```

### 💡 核心设计模式解读：

1. **TodoWrite vs Task System：两层计划系统，各司其职**
   | | TodoWrite (s05) | Task System (s12) |
   |---|---|---|
   | 定位 | 当前任务的执行清单 | 可恢复的任务系统 |
   | 存储 | 进程内存（退出即消失） | `.tasks/{id}.json`（跨会话持久） |
   | 依赖 | 无 | `blockedBy` 依赖图 |
   | 粒度 | Agent 自己的步骤 | 可被认领、追踪、解锁的任务 |
   | 协作 | 单人 | `owner` 字段支持多 Agent 认领 |

   TodoWrite 回答"我现在要干嘛"，Task System 回答"这个项目要干嘛"。前者是即时工作区，后者是持久化的目标管理。

2. **blockedBy：不是"建议先做"，是硬约束**
   ```python
   def can_start(task_id: str) -> bool:
       task = load_task(task_id)
       for dep_id in task.blockedBy:
           if not _task_path(dep_id).exists():       # 依赖文件不存在 → 阻塞
               return False
           if load_task(dep_id).status != "completed": # 依赖未完成 → 阻塞
               return False
       return True

   def claim_task(task_id: str, owner: str = "agent") -> str:
       if task.status != "pending":                   # 状态检查
           return f"Task {task_id} is {task.status}, cannot claim"
       if not can_start(task_id):                     # 依赖检查
           deps = [d for d in task.blockedBy if ... not completed]
           return f"Blocked by: {deps}"              # ← 硬拒绝
       task.owner = owner; task.status = "in_progress"
   ```
   `can_start` 在 `claim_task` 里被调用——不是"建议后再想想"，是**如果依赖未完成就直接拒绝认领**。缺失的依赖文件同样视为阻塞（不会因为文件找不到就跳过检查）。这是 DAG（有向无环图）约束：你只能从入度为 0 的节点开始。

3. **complete_task 的连锁反应：自动报告解锁的下游**
   ```python
   def complete_task(task_id: str) -> str:
       task.status = "completed"
       # 扫描所有 pending + 有 blockedBy 的任务，检查是否被本次完成解锁
       unblocked = [t.subject for t in list_tasks()
                    if t.status == "pending" and t.blockedBy and can_start(t.id)]
       msg = f"Completed {task.id} ({task.subject})"
       if unblocked:
           msg += f"\nUnblocked: {', '.join(unblocked)}"
   ```
   一个任务完成后，不自动处理下游——但主动告诉 Agent"这些任务现在可以做了"。下游任务仍然需要显式 `claim_task`，保持状态转换的显式控制。

4. **JSON 持久化：dataclass + asdict 零序列化代码**
   ```python
   @dataclass
   class Task:
       id: str; subject: str; description: str
       status: str; owner: str | None; blockedBy: list[str]

   def save_task(task: Task):
       _task_path(task.id).write_text(json.dumps(asdict(task), indent=2), encoding="utf-8")

   def load_task(task_id: str) -> Task:
       return Task(**json.loads(_task_path(task_id).read_text(encoding="utf-8")))
   ```
   不需要手写序列化/反序列化——`dataclass` + `asdict` + `**kwargs` 构造，4 行搞定读写。ID 用 `task_{timestamp}_{random}` 避免碰撞，教学版没有实现 CC 的 highwatermark 文件防重用。

5. **list_tasks 的输出设计：一目了然的状态+依赖**
   ```
   ○ task_001: 设计数据库表 [pending] (blockedBy: ...)
   ● task_002: 搭建 API [in_progress] [agent]
   ✓ task_003: 写单元测试 [completed]
   ```
   `run_list_tasks` 用 `○/●/✓` 图标区分 pending/in_progress/completed，owner 用方括号显示，blockedBy 直接展示。LLM 看到这个输出就能理解当前全局状态，不需要逐个 `get_task`。

6. **s11 → s12：加了 1 个 dataclass + 8 个函数 + 5 个工具**
   | 新增 | 说明 |
   |------|------|
   | `Task` dataclass | 数据结构（6 字段） |
   | `_task_path` / `create_task` / `save_task` / `load_task` | CRUD 基础 |
   | `list_tasks` / `get_task` | 查询 |
   | `can_start` | blockedBy 依赖全检查 |
   | `claim_task` / `complete_task` | 状态转换 |
   | 5 个工具函数 + schema | 映射到 TOOL_HANDLERS |

   s12 是教学骨架版，省略 s11 的完整 error recovery。真实 CC 中 `utils/tasks.ts` 和 `withRetry` 是独立层，自然组合互不耦合。
