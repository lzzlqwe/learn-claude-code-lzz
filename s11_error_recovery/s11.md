"API 会挂, 不是 bug 是常态" -- 三条恢复路径 + 指数退避 + 模型切换。

Harness 层: 容错 -- 网络抖、限流、超长, 每条都有应对。

```python
s11_error_recovery/code.py - 带错误恢复的 Agent

在 s10 的 prompt 组装之上，LLM 调用包裹在 try/except 中，按错误类型走三条恢复路径。
网络抖动、API 限流、输出截断、上下文超限——每种错误都有对应的恢复策略。

三条恢复路径:
  Path 1: max_tokens 截断 → 升级 8K→64K → continuation prompt（最多 3 次）
  Path 2: prompt_too_long → reactive_compact（尾 5 条）→ 重试（仅 1 次）
  Path 3: 429/529 → 指数退避 + jitter（最多 10 次），连续 529×3 切 fallback 模型

架构图：
    ┌────────────────────────────────────┐
    │  messages → get_system_prompt()     │
    └────────────────┬───────────────────┘
                     ▼
    ┌────────────────────────────────────┐
    │  try: with_retry(LLM call)         │  ← Path 3 在这里面（429/529 重试）
    │    Path 3: 429/529                 │
    │      → 指数退避 + jitter（×10）     │
    │      → 连续 529×3 → 切 fallback   │
    └────────────────┬───────────────────┘
                     ▼ (非瞬时错误向外抛)
    ┌────────────────────────────────────┐
    │  except:                           │
    │    Path 2: prompt_too_long         │  ← reactive_compact（1 次）
    │      → tail 5 + 占位消息           │
    │    other → 报错退出                  │
    └────────────────┬───────────────────┘
                     ▼ (成功拿到 response)
    ┌────────────────────────────────────┐
    │  Path 1: stop_reason == max_tokens │
    │    第一次 → 升级 64K（不追加截断输出）│
    │    再截断 → continuation prompt    │  ← "no apology, pick up mid-thought"
    │    超过 3 次 → 停止                 │
    └────────────────────────────────────┘

核心洞察："s01-s10 假设 API 调用一定成功。s11 承认现实——网络会抖、限流会发生、
         输出会被截断。不是 bug，是常态。每类错误都有恢复路径。"
```

### 💡 核心设计模式解读：

1. **三条路径按错误类型分流，不是一堆 if/elif**
   ```python
   try:
       response = with_retry(lambda: client.messages.create(...), state)
       # ↑ Path 3 在这里面：429/529 在 with_retry 内部自动重试
   except Exception as e:
       # ↑ Path 2：prompt_too_long 从 with_retry 抛出，外层捕捉
       if is_prompt_too_long_error(e):
           messages[:] = reactive_compact(messages)   # 保留尾 5 条
           continue
       # 无法恢复 → 报错退出

   # Path 1：API 调用成功了，但输出被截断
   if response.stop_reason == "max_tokens":
       if not escalated: max_tokens = 64000; continue  # 升级
       messages.append(continuation_prompt); continue   # 续写
   ```
   关键设计：`with_retry` 只处理它认识的错误类型（429/529），不认识的就向外抛。外层的 `except` 再处理 `prompt_too_long`，再往外就是不可恢复。这是**分层错误处理**——每层只负责自己能处理的。

2. **Path 1: max_tokens 两次升级，不浪费 token**
   ```
   8K 截断 → 升级到 64K 重试（不追加截断输出，因为 8K 的截断内容不完整）
   64K 截断 → 保留截断输出 + continuation prompt "no apology, pick up mid-thought"
   再截断 → 重复 continuation prompt（最多 3 次）
   ```
   第一次升级到 64K 时**不追加**截断输出——因为 8K 时的输出是半截的，追加上去等于把"半句话"喂给 LLM，反而干扰续写。64K 仍然截断时，输出至少是完整的段落，这时候才值得追加 + 续写提示。

3. **Path 2: prompt_too_long → reactive_compact → 1 次重试**
   ```python
   def reactive_compact(messages):
       tail = messages[-5:]   # 只保留最后 5 条
       return [{"role": "user",
                "content": "[Reactive compact] Earlier conversation trimmed."}, *tail]
   ```
   教学版简化：不调 LLM 生成摘要（s08 已覆盖），直接保留尾部 5 条 + 占位消息。真实 CC 会调 LLM 生成压缩摘要。只重试 1 次——如果压缩后还 `prompt_too_long`，说明 5 条消息本身就超限了，再重试也没用，直接报错退出。

4. **Path 3: 指数退避 + jitter + 模型切换**
   ```python
   def retry_delay(attempt):
       base = min(500ms * 2^attempt, 32s) / 1000     # 500ms, 1s, 2s, 4s, ... 32s
       jitter = random.uniform(0, base * 0.25)        # 加 25% 随机抖动
       return base + jitter

   # 529 连续 3 次 → 切换 FALLBACK_MODEL
   if consecutive_529 >= 3 and FALLBACK_MODEL:
       state.current_model = FALLBACK_MODEL
   ```
   两个关键细节：
   - **jitter**（随机抖动）：防止多个 Agent 同时重试形成"惊群效应"（thundering herd），25% 的随机化让重试时间错开
   - **529 模型切换**：连续 3 次 529（服务过载），说明当前模型扛不住了，切换到 fallback 模型。不是"每次都切"，而是"连续 3 次才切"——避免偶发过载就切模型

5. **RecoveryState：不是全局变量，是追踪对象**
   ```python
   class RecoveryState:
       has_escalated: bool        # max_tokens 只升级一次
       recovery_count: int         # continuation 最多 3 次
       consecutive_529: int       # 连续 3 次 → 切模型
       has_attempted_reactive_compact: bool  # 只压缩一次
       current_model: str          # 可能在 529 后被替换
   ```
   所有恢复状态集中在一个对象里，不散落成零散的全局变量。每轮循环重置 state（`state = RecoveryState()`），但同一轮内状态累积——比如 `has_escalated` 确保 8K→64K 只做一次。

6. **s10 → s11：加了 1 个类 + 4 个函数 + 3 个常量**
   | 新增 | 说明 |
   |------|------|
   | `RecoveryState` | 追踪升级/压缩/529/模型状态 |
   | `retry_delay(attempt)` | 指数退避 + jitter |
   | `with_retry(fn, state)` | 429/529 自动重试包装器 |
   | `is_prompt_too_long_error(e)` | 精确匹配多种错误消息 |
   | `reactive_compact(messages)` | 尾 5 条保留（教学简化版） |

   循环改动：LLM 调用从一行 `client.messages.create(...)` 变成 `with_retry(...)` + `try/except` + `max_tokens` 检查，约 40 行。但核心结构不变——工具执行、prompt 组装、context 更新全部继承 s10。
