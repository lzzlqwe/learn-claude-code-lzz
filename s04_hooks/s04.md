```python
s04_hooks/code.py - Hook 系统

    +----------+      +-------+      +-----------------------+      +------------------+
    |   用户    | ---> |  LLM  | ---> |     Hook 系统          | ---> |   工具执行        |

                          ^          | PostToolUse: 大输出警告  |      +------------------+
                          |          | Stop: 退出统计          |
                          +----------+                        |
                          返回工具执行结果 + hook 后处理          |
                                     +------------------------+

核心洞察："s03 在循环里调 check_permission()，s04 改为 trigger_hooks('PreToolUse', block)——循环只认识 hook，不认识任何具体逻辑。"
```

### 💡 核心设计模式解读：

1. **Hook 系统 = 注册表 + 触发器 + 回调约定**
   ```python
   # 注册表: 4 个事件，每个对应一个回调列表
   HOOKS = {"UserPromptSubmit": [], "PreToolUse": [], "PostToolUse": [], "Stop": []}

   # 注册: 往事件上挂回调
   def register_hook(event, callback):
       HOOKS[event].append(callback)

   # 触发: 遍历该事件的所有回调，遇到非 None 就停
   def trigger_hooks(event, *args):
       for callback in HOOKS[event]:
           result = callback(*args)
           if result is not None:   # 核心约定: 非 None = "拦下来了"
               return result
       return None
   ```
   整个系统的核心就这 3 个东西。回调返回 `None` = 放行（不干预），返回字符串 = 阻断（字符串就是原因，会直接作为工具结果告知 LLM）。这和 s02/s03 的"新工具只需加一行映射"是同一个扩展哲学。

2. **s03 → s04：从硬编码到可注册**
   对比 s03 和 s04 的 agent_loop：
   ```python
   # s03: check_permission() 硬编码在循环里
   for block in response.content:
       if not check_permission(block):  # ← 每次加新功能都要改这里
           continue
       output = execute(block)

   # s04: trigger_hooks() 替代硬编码
   for block in response.content:
       if trigger_hooks("PreToolUse", block):  # ← 循环只认识 hook
           continue
       output = execute(block)
       trigger_hooks("PostToolUse", block, output)  # ← 多了一行 post hook
   ```
   s03 的 `check_permission()` 逻辑原封不动搬进 `permission_hook` 回调，通过 `register_hook("PreToolUse", permission_hook)` 挂上去。s04 还顺手加了 PostToolUse 点（大输出警告），但循环只多了一行 `trigger_hooks`，而不是一坨 if/elif。

3. **4 个事件覆盖的完整 Agent 生命周期**
   ```
   用户输入 → [UserPromptSubmit] → LLM 调用 → [PreToolUse] → 工具执行 → [PostToolUse] → 回传结果 → ... → 循环将退 → [Stop]
   ```
   | 事件 | 触发时机 | 教学版回调 | 返回值作用 |
   |------|---------|-----------|-----------|
   | UserPromptSubmit | 用户输入后、LLM 前 | context_inject_hook（打印工作目录） | 未被使用 |
   | PreToolUse | 工具执行前 | permission_hook（权限检查）、log_hook（日志） | 非 None → 阻断执行 |
   | PostToolUse | 工具执行后 | large_output_hook（大输出警告） | 未被使用 |
   | Stop | 循环退出前 | summary_hook（统计工具调用次数） | 非 None → 强制续跑 |

   **教学简化**：Claude Code 实际有 27 个 hook 事件和 14 个返回字段。教学版只讲 4 个核心事件，覆盖一个 agent cycle 的关键节点，其余的都是同样的机制。

4. **Stop hook 的巧妙设计：不让 Agent 停机**
   ```python
   if response.stop_reason != "tool_use":
       force = trigger_hooks("Stop", messages)
       if force:                                     # hook 返回了字符串
           messages.append({"role": "user", "content": force})  # 当成新用户消息注入
           continue                                  # 不让退出，回到循环顶部
       return                                         # force 为 None，正常退出
   ```
   Stop hook 返回非 None → 注入一条"用户消息" → `continue` 回到 LLM 调用。教学版中 `summary_hook` 总是返回 `None`（放行），但机制留好了——以后写个"如果没写测试就续跑"的 hook 直接注册即可。

5. **s01→s04 的进化脉络**
   | 版本 | 核心新增 | 循环改动 | 扩展方式 |
   |------|---------|---------|---------|
   | s01 | 1 个 bash 工具 + agent_loop | — | 改循环 |
   | s02 | 5 个工具 + Dispatch Map | 工具执行改为查表 | 加工具不改循环 |
   | s03 | 三道闸门权限管道 | +1 行 check_permission() | 加权限要改循环 |
   | s04 | Hook 注册表系统 | 改成 trigger_hooks() | **加什么都不改循环** |

   每次进化都在让核心循环更稳定、更干净。到了 s04，循环本身变成了一个"骨架"——它只负责调度，具体行为全部交给 hook。这就是**发布-订阅模式**在 Agent 架构中的经典应用。
