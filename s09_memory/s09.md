"压缩会丢细节, 要有一层不丢的" -- 文件仓库 + 索引 + 按需加载，跨压缩、跨会话。

Harness 层: 记忆 -- 跨压缩、跨会话的知识积累。

### 快速理解

```python
def agent_loop:
   > 加载记忆：根据最近对话选相关记忆（LLM side-query，失败降级关键词匹配）relevant_memories
   while true:
      > 重建 SYSTEM：每轮拉取最新 MEMORY.md 索引注入 system prompt
      > 保存快照：压缩前保存原始消息（留给后续 记忆提取 使用）
      > s08压缩管线：budget → snip → micro → [超限则 auto compact]
      > 记忆拼接：将相关记忆 relevant_memories 拼到当前 user turn 开头，不污染原始 messages（不破坏 cache）
      > LLM 调用 + 工具执行
   退出循环后：
   > 提取记忆：用LLM，提取前会先检查已有记忆，避免重复。
   > 整理记忆：用LLM，低频合并去重
```

### 💡 核心设计模式解读：

1. **为什么压缩之后还要 Memory？**
   s08 的 autoCompact 把 5000 条消息压缩成一段摘要。细节注定丢失："用 tab 不要用空格"可能简化成"用户有代码风格偏好"。更糟的是，新会话连摘要都没了。

   Memory 和压缩管线的分工：
   | | Compaction (s08) | Memory (s09) |
   |---|---|---|
   | 持久性 | 单会话（压缩后在摘要中） | 跨会话（文件系统持久） |
   | 粒度 | 全量对话 → 一段摘要 | 每条偏好/事实 → 一个 .md 文件 |
   | 触发 | 每轮自动 | 每轮结束时提取 + 定期整理 |
   | 丢不丢 | 有损压缩 | 无丢（文件完整保留） |

2. **存储：YAML frontmatter + 索引文件**
   ```python
   # .memory/user-preference-tabs.md
   ---
   name: user-preference-tabs
   description: User prefers tabs for indentation
   type: user
   ---
   User prefers using tabs, not spaces. **Why:** Consistency with existing codebase.
   ```

   ```markdown
   # .memory/MEMORY.md（索引，注入 SYSTEM prompt）
   - [user-preference-tabs](user-preference-tabs.md) — User prefers tabs for indentation
   ```

   每个记忆 = 一个 `.md` 文件，YAML frontmatter 存 `name/description/type`。写新记忆时 `write_memory_file()` 自动调用 `_rebuild_index()` 重建 `MEMORY.md`。索引 ≤ 200 行，注入 SYSTEM 的开销可控。

3. **加载：两条路径，分"常驻"和"按需"**
   ```
   路径一（常驻）: MEMORY.md 索引 → SYSTEM prompt → 每轮都带（但只有一行描述，token 少）
   路径二（按需）: select_relevant_memories() → LLM 根据最近对话选 → load_memories() 注入 user turn
   ```

   路径二用了 **LLM 自己来选**（不是 embedding 向量相似度）：
   ```python
   def select_relevant_memories(messages, max_items=5):
       # 把 name + description 列成清单
       catalog = "\n".join(f"{i}: {f['name']} — {f['description']}")
       # 发给 LLM 轻量 side-query："选出真正有用的（最多 5 个）"
       response = client.messages.create(...)
       # 失败降级: 关键词匹配 name + description
   ```
   Claude Code 源码也用 Sonnet 本身做这个选择（而非 embedding），因为 Sonnet 能理解语义关联，不需要建向量库。

4. **写入：散落在对话中的偏好，自动提取**
   用户很少会说"记住这个"。偏好散落在正常对话中："用 tab 比空格好"、"以后都用单引号"。

   ```python
   # agent_loop 中，模型停止且无 tool_use 时触发
   if response.stop_reason != "tool_use":
       extract_memories(pre_compress)   # ← 从压缩前快照提取（保证完整）
       consolidate_memories()
       return
   ```

   提取前先列出已有记忆，避免重复。提取 prompt 让 LLM 返回 `{name, type, description, body}` 的 JSON。只有"之前没存过的新信息"才会写文件。提取用的是 `pre_compress`（压缩前的消息快照），保证不丢细节。

5. **整理（Dream）：≥10 条时自动去重合并**
   ```python
   CONSOLIDATE_THRESHOLD = 10
   def consolidate_memories():
       files = list_memory_files()
       if len(files) < CONSOLIDATE_THRESHOLD:
           return  # 太少，不值得
       # 把所有记忆发 LLM → 去重/合并矛盾/删除过时
       # 删旧文件 → 写新文件 → 重建索引
   ```
   记忆会积累——20 次"记住"后可能有重复矛盾。Dream 每 ≥10 条触发一次，LLM 做一次批量整理。CC 源码有四层门控（时间/扫描/会话/锁），教学版简化为文件数阈值。

6. **四类记忆，四种用途**
   | 类型 | 回答的问题 | 示例 |
   |------|-----------|------|
   | user | 用户是谁 | "用 tab 不用空格" "我是后端工程师" |
   | feedback | 怎么做事 | "别 mock 数据库" "测试要用真实 DB" |
   | project | 正在发生什么 | "auth 重写是合规驱动的" "周四冻结合并" |
   | reference | 东西在哪找 | "pipeline bug 在 Linear INGEST" |

7. **s08 → s09：加了 5 个函数 + 2 个存储目录，循环没膨胀**
   | 新增 | 说明 |
   |------|------|
   | `write_memory_file` + `_rebuild_index` | 写入一条记忆并重建索引 |
   | `read_memory_index` + `read_memory_file` | 读索引 / 读单条内容 |
   | `list_memory_files` | 列出所有记忆元数据 |
   | `select_relevant_memories` | LLM 选（+ 关键词降级） |
   | `load_memories` | 包装为 `<relevant_memories>` 标签 |
   | `extract_memories` | 每轮结束后从对话提取 |
   | `consolidate_memories` | ≥10 条时整理去重 |

   循环只多了 3 行：`load_memories()` 在入口、`extract_memories()` + `consolidate_memories()` 在出口。
