### 💡 核心设计模式解读：

1. **同步等 vs 后台跑：不只是性能问题**
   读 `README.md` 是毫秒级，`git status` 是秒级，等一等没问题。但 `pip install torch` 要 10 分钟，`npm run build` 要 3 分钟——Agent 干等就是纯浪费。

   | | 同步 (s12) | 后台 (s13) |
   |---|---|---|
   | 慢操作 | Agent 干等 | 后台线程执行 |
   | Agent 空闲 | 是（等 bash 返回） | 否（继续下轮 LLM 调用） |
   | 结果 | 立即返回 | 下轮作为通知注入 |
   | 判断标准 | — | 模型显式请求 + 启发式兜底 |

   关键不是"快"，而是"不阻塞"。`start_background_task` 做完三件事就立刻返回占位符——注册 bg_id、启动线程、返回 `"[Background task bg_0001 started]"`——主循环继续跑，不等 worker 完成。

2. **should_run_background：两条判断路径**
   ```python
   def should_run_background(tool_name, tool_input) -> bool:
       # 路径 1: 模型显式请求（优先级最高）
       if tool_input.get("run_in_background"):
           return True
       # 路径 2: 关键词启发式（模型没说时自动判断）
       return is_slow_operation(tool_name, tool_input)

   def is_slow_operation(tool_name, tool_input) -> bool:
       if tool_name != "bash": return False
       slow_kw = ["install", "build", "test", "deploy", "compile",
                  "docker build", "pip install", "npm install", ...]
       return any(kw in cmd for kw in slow_kw)
   ```
   路径 1 让 LLM 自己决定——bash 工具的 schema 里加了 `run_in_background: boolean` 参数，模型可以根据语义判断"这个命令会很久"。路径 2 是兜底——如果模型忘了传参数，关键词匹配也能自动识别慢操作。两条路径叠加，不漏不误。

3. **线程安全：不用 queue，用 Lock + dict**
   ```python
   background_lock = threading.Lock()
   background_tasks: dict = {}   # 主线程写(注册) + worker 线程写(status)
   background_results: dict = {} # worker 线程写 + 主线程读

   # worker 线程: 写结果
   with background_lock:
       background_tasks[bg_id]["status"] = "completed"
       background_results[bg_id] = result

   # 主线程: 读结果
   with background_lock:
       ready_ids = [bid for bid in background_tasks if ... == "completed"]
   ```
   用 `threading.Lock` 而不是 `queue.Queue`：教学版只有主线程 + 几个 daemon worker，dict + lock 足够简单直观。daemon 线程不会阻塞进程退出——主线程结束，所有 worker 自动终止。

4. **collect_background_results：pop + 格式化**
   ```python
   def collect_background_results():
       # 1. 上锁收集已完成的 bg_id
       ready_ids = [bid for bid in background_tasks if ... == "completed"]
       # 2. 逐个 pop（取走就没了，不会重复注入）
       for bg_id in ready_ids:
           task = background_tasks.pop(bg_id)   # ← pop，不重复
           output = background_results.pop(bg_id)
       # 3. XML 格式通知
       notifications.append(
           f"<task_notification>\n"
           f"  <task_id>{bg_id}</task_id>\n"
           f"  <command>{task['command']}</command>\n"
           f"  <summary>{output[:200]}</summary>\n"
           f"</task_notification>")
   ```
   关键细节：`pop()` 而不是 `get()`。通知取走后立即删除，不会在后续轮次重复注入。XML 格式匹配真实 CC 的通知格式，LLM 能理解 `<task_notification>` 的语义。

5. **通知与工具结果合并在同一条消息中**
   ```python
   user_content = []
   # 先放后台完成通知
   for notif in bg_notifications:
       user_content.append({"type": "text", "text": notif})
   # 再放本轮工具结果
   user_content.extend(results)
   messages.append({"role": "user", "content": user_content})
   ```
   通知和结果（实际中工具结果在前，通知在后）合并在**一条** `user` 消息里发给 LLM，而不是分成两条。LLM 在一轮里同时看到"后台跑完了"和"当前工具的输出"，可以综合决策。如果能并行跑的后台任务互相独立，LLM 可以同时发起多个。

6. **s12 → s13：加了 1 个锁 + 2 个字典 + 5 个函数**
   | 新增 | 说明 |
   |------|------|
   | `background_tasks` / `background_results` | 生命周期追踪 + 结果存储 |
   | `background_lock` | 线程安全 |
   | `is_slow_operation` | 关键词启发式 |
   | `should_run_background` | 显式请求 + 启发式 |
   | `start_background_task` | daemon 线程派发 + 占位符返回 |
   | `collect_background_results` | pop + XML 格式化 |
   | bash schema 新增 `run_in_background` | 模型可显式请求后台 |

   主循环改动：工具执行前插入 `should_run_background` 分流，工具执行后插入 `collect_background_results` 通知收集。
