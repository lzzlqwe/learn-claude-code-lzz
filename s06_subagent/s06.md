"大任务拆小, 每个小任务干净的上下文" -- 子 Agent 用独立 messages[]，不污染主对话。

Harness 层: 子 Agent -- 上下文隔离, 注意力不漂移。

```python
s06_subagent/code.py - 带有子 Agent 派生能力的 Agent

在 s05 的 todo_write + hook 系统之上，新增一个 task 工具。调用它时 spawn 一个子 Agent，
拥有全新的 messages[]，跑自己的循环，结束后只把摘要文本回传给主 Agent。

架构图：
    Parent Agent                           Subagent
    +------------------+                  +------------------+
    | messages=[...]   |                  | messages=[task]  | <-- 干净上下文
    |                  |   dispatch       |                  |
    | tool: task       | ---------------> | own while loop   |
    |   prompt="..."   |                  |   bash/read/...  |
    |                  |   summary only   |   (max 30 turns) |
    | result = "..."   | <--------------- | return last text |
    +------------------+                  +------------------+
                                               |
                 中间过程全部丢弃 ← ← ← ← ← ← ← ← ←
         （文件系统副作用保留，对话上下文不保留）

    子 Agent 工具: bash/read/write/edit/glob（无 task/无 todo_write —— 不递归不计划）
    子 Agent 同样过 PreToolUse hook —— 上下文隔离不代表权限隔离

核心洞察："子 Agent 的所有中间对话——读了哪些文件、跑了哪些命令、修了几轮 bug——全部丢弃。
         回到主 Agent 的只有一句结论。这就是上下文隔离。"
```

### 💡 核心设计模式解读：

1. **为什么需要子 Agent？上下文隔离解决什么问题？**
   Agent 修一个 bug，读了 30 个文件追踪调用链，messages 涨到 120 条——其中大部分和"修 bug"这个最终目标无关。这些中间过程占着上下文，让 Agent 越来越健忘。

   子 Agent 解决的就是这个问题：像一个独立的子进程，给它一个干净的消息列表，让它专心做一件事。做完之后，子进程的"思考过程"全部丢弃，只有结论传回主 Agent。

2. **spawn_subagent：独立循环 + 30 轮锁 + 摘要返回**
   ```python
   def spawn_subagent(description: str) -> str:
       messages = [{"role": "user", "content": description}]  # 全新 messages
       for _ in range(30):  # 安全上限
           response = client.messages.create(
               model=MODEL, system=SUB_SYSTEM,  # 独立 SYSTEM prompt
               messages=messages, tools=SUB_TOOLS, ...
           )
           # ... 同样的 while 循环、hook、执行、回传 ...
       return extract_text(messages[-1]["content"])  # 只返回文本结论
   ```
   子 Agent 内部也是一个完整的 agent_loop——调 LLM → 过 hook → 执行工具 → 回传结果。和父 Agent 完全一样，区别有三：
   - **SUB_SYSTEM**：独立的 system prompt，告诉它"直接完成任务，不要委派"
   - **SUB_TOOLS**：只有 5 个基础工具，**没有 task**——防止递归派生
   - **返回摘要**：`extract_text(last_message)`，而不是整个 messages 列表

3. **防止递归：不是限制深度，是从能力上禁止**
   ```python
   # 父 Agent 工具: 7 个（含 task）
   TOOL_HANDLERS = { ..., "task": spawn_subagent }

   # 子 Agent 工具: 5 个（无 task）
   SUB_TOOLS = [bash, read_file, write_file, edit_file, glob]
   # 无 "task" 工具 —— 子 Agent 根本不知道 task 这个东西存在
   ```
   子 Agent 的 `SUB_TOOLS` 列表里根本没有 `task` 这个工具。LLM 在子 Agent 的循环里看不到 task 工具，自然不可能调用它。这是**能力层面的硬限制**，不需要额外代码判断"当前是第几层"。

4. **权限统一：子 Agent 的工具调用也走 hook**
   ```python
   # 子 Agent 内部循环
   for block in response.content:
       if block.type == "tool_use":
           blocked = trigger_hooks("PreToolUse", block)  # ← 同样的 hook
           ...
   ```
   上下文隔离不代表权限隔离。子 Agent 的 bash 调用同样经过 `DENY_LIST` 检查，如果尝试 `rm -rf /`，照样被 Gate 1 拦截。这和 s04 的 hook 系统完全兼容——hook 注册一次，父/子 Agent 都生效。

5. **s05 → s06：只加了一个工具，Dispatch Map 完全不用动**
   | 新增 | 改动量 | 改动位置 |
   |------|--------|---------|
   | `spawn_subagent()` | +45 行 | 独立函数，内含完整子循环 |
   | `SUB_TOOLS` | +10 行 | 独立工具列表（无 task） |
   | `SUB_HANDLERS` | +2 行 | 子 Agent 分发映射 |
   | `task` schema | +4 行 | 追加到 TOOLS 列表 |
   | `TOOL_HANDLERS["task"]` | 1 行 | 一行映射 |
   | `extract_text()` | +4 行 | 辅助函数 |

   主循环完全没变。子 Agent 的循环和父 Agent 结构一致，只是上下文独立 + 工具受限 + 返回摘要。
