"上下文总会满, 要有办法腾地方" -- 四层压缩管线, 便宜的先跑贵的后跑。

Harness 层: 压缩 -- 干净的记忆, 无限的会话。

```python
s08_context_compact/code.py - 带四层上下文压缩管线的 Agent
```

### 💡 核心设计模式解读：

1. **为什么要四层？只用 LLM 摘要（L4）不行吗？**
   L4 最强大但最贵——每次触发花 1 次 LLM API 调用。如果每轮都跑 L4，Agent 跑 30 轮，其中 29 轮是"摘要 LLM 调用"而非"干活 LLM 调用"。前 3 层的目的就是**尽可能用免费手段撑住，L4 只在不得已时才触发**。

   | 层 | 成本 | 手段 |
   |----|------|------|
   | L3 budget | 0 API | 大结果落盘（超出 200KB 时） |
   | L1 snip | 0 API | 裁剪中间消息（超出 50 条时） |
   | L2 micro | 0 API | 旧 tool_result 替换为占位符（超出 3 条保留） |
   | L4 auto | **1 API** | LLM 全文摘要 |

2. **执行顺序为什么是 L3→L1→L2，不能换？**
   ```python
   messages[:] = tool_result_budget(messages)    # L3 先跑
   messages[:] = snip_compact(messages)          # L1
   messages[:] = micro_compact(messages)         # L2 最后
   ```
   L3（budget）必须在 L2（micro）前面跑——因为 L2 会把旧的大 tool_result 替换成一行 `"[Earlier tool result compacted.]"` 占位符。如果 L2 先跑，那些 30KB 的旧文件内容就变成一行了，L3 想落盘也没东西可落。这和 Claude Code 源码中 `applyToolResultBudget` 跑在最前面的逻辑一致。

3. **L1 snip: 裁消息 ≠ 丢上下文**
   ```python
   def snip_compact(messages, max_messages=50):
       if len(messages) <= max_messages: return messages
       keep_head, keep_tail = 3, max_messages - 3
       return messages[:keep_head] + [
           {"role": "user", "content": f"[snipped {snipped} messages]"}
       ] + messages[-keep_tail:]
   ```
   不是"丢了中间忘了开头"——保留头部 3 条（初始任务定义）和尾部 47 条（当前工作）。中间插入一条占位消息告诉 LLM"中间已经裁掉了"，LLM 知道它缺少这块上下文，不会基于不存在的信息做出错误推断。

4. **L2 micro: 旧结果换占位符 + 最近 3 条白名单**
   ```python
   def micro_compact(messages):
       tool_results = collect_tool_results(messages)
       if len(tool_results) <= KEEP_RECENT: return messages  # ≤3 → 不碰
       for _, _, block in tool_results[:-KEEP_RECENT]:       # 旧的结果
           if len(block.get("content", "")) > 120:
               block["content"] = "[Earlier tool result compacted. Re-run if needed.]"
       return messages
   ```
   关键设计：保留最近 3 条 tool_result 不动。Agent 刚读的文件内容可能马上要用，丢掉会导致它重新读文件（浪费 API 调用）。旧的结果换成占位符，LLM 如果发现需要，可以重新执行工具获取最新数据。

5. **L4 + reactive: 两条摘要路径，一主动一被动**
   ```
   L4 auto_compact（主动）: token 超阈值 → 备份 → LLM 摘要 → 替换 messages
   reactive_compact（被动）: API 报 prompt_too_long → 备份 → LLM 摘要 → 摘要 + 尾5条
   ```
   区别：L4 只保留摘要；reactive_compact 保留摘要 + 最近 5 条消息。"尾 5 条"保留的是触发报错那一轮的具体上下文，让 LLM 知道"刚才到底干了什么导致超长"。

6. **compact 工具：Agent 可以主动压缩**
   ```python
   if block.name == "compact":
       messages[:] = compact_history(messages)  # 直接替换整个 messages
       break                                     # 结束当前轮
   ```
   这不是一个普通工具——它的"返回值"不是文本，而是直接替换整个 messages 列表。Agent 觉得自己跑太久了、上下文快满了，可以主动调用 `compact` 做一次全量摘要。其他工具返回内容是"追加到消息列表后面"，compact 是"整个列表换掉"。

7. **s07 → s08：加了 5 个函数 + 1 个工具，循环不膨胀**
   | 新增 | 改动量 | 说明 |
   |------|--------|------|
   | `snip_compact()` | +8 行 | L1: 裁剪中间消息 |
   | `micro_compact()` + `collect_tool_results()` | +15 行 | L2: 旧结果占位 |
   | `tool_result_budget()` + `persist_large_output()` | +20 行 | L3: 大结果落盘 |
   | `compact_history()` + `summarize_history()` + `write_transcript()` | +20 行 | L4: LLM 摘要 |
   | `reactive_compact()` | +4 行 | 应急压缩 |
   | `compact` schema | +3 行 | Agent 可主动触发 |

   循环本体的改动：LLM 调用前插入 4 行 `messages[:] = ...` + 1 个 `if/except`。所有压缩逻辑是独立的纯函数，互不耦合。
